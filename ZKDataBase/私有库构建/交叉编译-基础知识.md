本文档是 [[交叉编译-主文档]] 的配套基础知识文档。

## 1. ABI（Application Binary Interface）

ABI 规定编译后的二进制代码在运行时如何交互：

- 函数参数通过哪些寄存器传递
- 返回值放在哪个寄存器
- 栈如何对齐
- 结构体成员如何排列（填充规则）
- 符号名称如何修饰

**API 和 ABI 的对比：**

| | API | ABI |
|---|---|---|
| 层面 | 源码级约定 | 二进制级约定 |
| 定义者 | 头文件中的函数声明 | CPU 架构 + 操作系统 + 编译器 |
| 例子 | `int add(int a, int b)` | `a→r0, b→r1, 返回值→r0`（ARM） |
| 不兼容的后果 | 编译不过（类型错误） | 链接不过或运行时崩溃 |

交叉编译工具链名中的 `gnueabihf` 拆解：
- `gnu`：使用 GNU C 库（glibc）
- `eabi`：嵌入式 ABI（Embedded ABI）
- `hf`：硬件浮点（hard float）—— 浮点参数通过 FPU 寄存器传递

## 2. 交叉编译工具链（Cross-compilation Toolchain）

一套在不同架构上运行的编译工具集合。核心组件：

```
arm-linux-gnueabihf-gcc      ← C 编译器前端（调用 cc1 + as + ld）
arm-linux-gnueabihf-g++      ← C++ 编译器前端
arm-linux-gnueabihf-ar       ← 静态库打包工具
arm-linux-gnueabihf-ld       ← 链接器
arm-linux-gnueabihf-objdump  ← 反汇编工具
arm-linux-gnueabihf-readelf  ← ELF 分析工具
arm-linux-gnueabihf-strip    ← 符号裁剪工具
```

安装后这些工具放在 `/usr/bin/` 下，和本机的 gcc 并存。通过前缀区分——不带前缀的是本地工具链，带 `arm-linux-gnueabihf-` 的是交叉工具链。

## 3. sysroot（目标系统根文件系统镜像）

sysroot 是目标 ARM 系统根文件系统在 PC 上的拷贝——包含目标架构的所有头文件和库文件。

```
/path/to/arm-rootfs/
├── usr/
│   ├── include/               # 目标架构的 C/C++ 头文件
│   │   ├── stdio.h
│   │   ├── openssl/
│   │   │   └── ssl.h
│   │   └── mosquitto.h
│   ├── lib/                   # 目标架构的动态库和静态库
│   │   ├── libssl.so
│   │   ├── libcrypto.a
│   │   └── libmosquitto.so
│   └── local/
│       ├── include/
│       └── lib/
└── lib/
    └── ld-linux-armhf.so.3    # 目标架构的动态链接器
```

获取 sysroot 的方式：
- 从 ARM 板子上直接拷贝整个根文件系统（`rsync -av root@board:/ /path/to/arm-rootfs/`）
- 使用交叉编译工具链自带的 sysroot（通常在 `/usr/arm-linux-gnueabihf/` 下）
- 用构建系统（Buildroot、Yocto）生成标准化的 sysroot

CMake 中通过 `CMAKE_SYSROOT` 指向 sysroot 路径，编译器自动使用其中的头文件和库文件。

## 4. `CMAKE_FIND_ROOT_PATH_MODE` 四种模式

CMake 在交叉编译时，对查找路径的控制通过这四个变量实现：

```cmake
set(CMAKE_FIND_ROOT_PATH_MODE_PROGRAM NEVER)
set(CMAKE_FIND_ROOT_PATH_MODE_LIBRARY ONLY)
set(CMAKE_FIND_ROOT_PATH_MODE_INCLUDE ONLY)
set(CMAKE_FIND_ROOT_PATH_MODE_PACKAGE ONLY)
```

| 变量 | 可选值 | 含义 |
|------|--------|------|
| `PROGRAM` | NEVER / ONLY / BOTH | `find_program` 的搜索范围 |
| `LIBRARY` | NEVER / ONLY / BOTH | `find_library` 的搜索范围 |
| `INCLUDE` | NEVER / ONLY / BOTH | `find_path` 和 `find_file` 的搜索范围 |
| `PACKAGE` | NEVER / ONLY / BOTH | `find_package` 的搜索范围 |

各选项：
- **NEVER**：只在主机（PC）路径搜索，不管 sysroot
- **ONLY**：只在 sysroot 中搜索
- **BOTH**：两边都搜，主机优先

`PROGRAM NEVER` 的原因是 cmake、make、pkg-config 这类**构建时运行的工具**必须用 PC 版本——它们跑在 PC 上而不是 ARM 上。

`LIBRARY/INCLUDE ONLY` 的原因是链接和编译需要的 .so 和 .h 必须是 ARM 版本的——错拿 PC 上的 x86-64 版本会导致链接错误或运行时崩溃。

## 5. `file` 命令——验证产物架构

交叉编译完成后最常用的验证工具：

```bash
file hello.o
# 输出示例（ARM 32位）：
# hello.o: ELF 32-bit LSB relocatable, ARM, EABI5 version 1 (SYSV)

file hello.o
# 输出示例（x86-64）：
# hello.o: ELF 64-bit LSB relocatable, x86-64, version 1 (SYSV)

file libhello.so
# 输出示例（ARM 动态库）：
# libhello.so: ELF 32-bit LSB shared object, ARM, EABI5 version 1 (SYSV)
```

`file` 通过读取文件头魔数判断架构，不依赖扩展名。

## 6. 交叉编译产生的典型错误

### 错误一：忘记用交叉工具链

```bash
gcc -c hello.c -o hello.o    # 用了本地 gcc
file hello.o
# hello.o: ELF 64-bit LSB relocatable, x86-64  ← 架构不对
```

### 错误二：ABI 混用

```bash
# 库用 hf 编译
arm-linux-gnueabihf-gcc -shared -o libmyuart.so uart.o

# 主程序用 sf 编译
arm-linux-gnueabi-gcc main.c -lmyuart -o main
# 链接时报浮点符号不匹配
```

### 错误三：sysroot 路径配置错误

```bash
# CMAKE_SYSROOT 指向一个不存在或不完整的目录
# CMake 输出：
# -- Check size of void* - done
# CMake Error: Could NOT find Threads (missing: Threads_FOUND)
# 原因：CMAKE_SYSROOT 中没有 pthread 库
```

### 错误四：ldd 对 ARM 二进制无效

```bash
ldd ./test_uart
# not a dynamic executable
# 因为 ldd 只能解析本机架构的 ELF
```

交叉编译场景用 `readelf` 替代：

```bash
arm-linux-gnueabihf-readelf -d ./test_uart | grep NEEDED
# 0x00000001 (NEEDED)  Shared library: [libmyuart.so]
# 0x00000001 (NEEDED)  Shared library: [libc.so.6]
```
