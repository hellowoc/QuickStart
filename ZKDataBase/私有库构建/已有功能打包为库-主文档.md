# 已有功能打包为库-主文档

## 本文主线

假设你手上已经有一组能正常工作的 .c 和 .h 文件——它们实现了某个独立功能（比如一个串口通信模块、一个 JSON 解析器、一个环形缓冲区）。本文的目标是：把这组文件从"散落在工程里的源码"变成"一个可以复用的独立库（.a + .so）"，然后在其他项目中使用它。

全文按实际操作顺序展开，每一步都说明**做什么**和**为什么这样做**。

---

# 第一步：审视现状——你手里有什么

## 假设场景

你有一个串口通信模块，当前目录结构：

```
my_uart/
├── uart.c          # 串口初始化、发送、接收
├── uart.h          # 公开函数声明
├── uart_internal.h # 内部用到的结构体、常量
├── ringbuffer.c    # 环形缓冲区（uart.c 内部使用）
└── ringbuffer.h    # 环形缓冲区声明（内部使用）
```

使用者（main.c）当前的用法是直接把这两个 .c 文件加入到自己的编译列表里：

```bash
gcc main.c uart.c ringbuffer.c -o main
```

## 现状的问题

1. **每次新项目都要拷贝源码**——uart.c、ringbuffer.c 到处复制，版本混乱
2. **编译浪费时间**——每次编译都要重新编译 uart.c 和 ringbuffer.c
3. **没有清晰的接口边界**——uart_internal.h 和 ringbuffer.h 也被使用者看到，容易误用内部函数
4. **不能独立测试**——模块没有独立于主程序的测试

## 目标

把上述模块变成：

```
my_uart/
├── include/mylib/          # 使用者只需要看这个目录
│   └── uart.h              # 仅包含公开 API
├── src/                    # 使用者不需要关心
│   ├── uart.c
│   ├── uart_internal.h
│   ├── ringbuffer.c
│   └── ringbuffer.h
├── CMakeLists.txt          # 构建脚本
└── test/
    └── test_uart.c         # 库的独立测试
```

编译产出：`libmyuart.a` 和 `libmyuart.so`。

使用者只需两行：

```bash
gcc main.c -I./include -L./lib -lmyuart -o main
```

# 第二步：整理目录结构

## 做什么

把你现有的文件分到两个目录：

- `include/mylib/` —— 放使用者需要的 .h 文件
- `src/` —— 放所有 .c 文件 + 内部 .h 文件

```
my_uart/
├── include/
│   └── mylib/
│       └── uart.h           ← 从原位置移过来
└── src/
    ├── uart.c               ← 从原位置移过来
    ├── uart_internal.h      ← 从原位置移过来
    ├── ringbuffer.c         ← 从原位置移过来
    └── ringbuffer.h         ← 从原位置移过来
```

## 为什么这样做

**原因一：接口和实现分离。** 头文件是"合同"——约定库提供什么功能；源文件是"工厂"——怎么实现这些功能。把合同放在 include/ 里，使用者和库开发者读同一份合同，不会出现实现细节的泄露。

**原因二：安装时只暴露头文件。** `cmake --install` 会把 include/ 拷到系统目录，src/ 不会。这意味着使用你库的人，在 `/usr/include/mylib/uart.h` 下只能看到公开 API，看不到 `uart_internal.h` 和 `ringbuffer.h`。

**原因三：include/mylib/ 的子目录是命名空间。** 使用者写 `#include <mylib/uart.h>` 而不是 `#include "uart.h"`。当另一个库也有 uart.h 时不会冲突。

# 第三步：检查并调整头文件

## 做什么

打开 uart.h，确认它只包含**使用者真正需要的声明**：

```c
// include/mylib/uart.h —— 公开头文件
#ifndef MYLIB_UART_H
#define MYLIB_UART_H

#include <stdint.h>

// ============================================
// 公开：使用者需要调用的函数
// ============================================

// 初始化串口，返回文件描述符
int uart_init(const char *device, int baudrate);

// 发送数据，返回实际发送字节数
int uart_send(int fd, const uint8_t *data, int len);

// 接收数据，返回实际接收字节数
int uart_recv(int fd, uint8_t *buf, int max_len);

// 关闭串口
void uart_close(int fd);

#endif
```

把内部声明移到 `src/uart_internal.h`：

```c
// src/uart_internal.h —— 内部头文件，不对外
#ifndef UART_INTERNAL_H
#define UART_INTERNAL_H

#include <termios.h>
#include "ringbuffer.h"

#define UART_BUF_SIZE 4096

// 内部函数——使用者不需要知道
int _uart_set_attr(int fd, int baudrate);
int _uart_flush(int fd);

// 内部状态——使用者不需要看到
typedef struct {
    int fd;
    struct termios old_tios;
    ringbuffer_t rx_buf;
    ringbuffer_t tx_buf;
} uart_ctx_t;

#endif
```

## 为什么这样做

**原因一：控制暴露范围。** 如果 `uart_ctx_t` 放在公开头文件中，使用者可能直接访问它的成员——如 `ctx.fd`。这产生了对内部结构的依赖。将来你把 `fd` 改名或删掉，使用者的代码就编译不过了。不暴露内部结构，你改内部实现时不影响使用者。

**原因二：减少编译依赖。** 公开头文件不需要 `#include "ringbuffer.h"` 和 `#include <termios.h>`，使用者编译时不会拉入这些和业务无关的头文件，编译更快，也不会因为 ringbuffer.h 的修改而重新编译使用者代码。

# 第四步：编写 CMakeLists.txt

## 做什么

在 `my_uart/` 根目录创建 CMakeLists.txt：

```cmake
cmake_minimum_required(VERSION 3.10)
project(myuart VERSION 1.0.0 LANGUAGES C)

# ============================================================
# 收集源文件
# ============================================================
set(SOURCES
    src/uart.c
    src/ringbuffer.c
)

# ============================================================
# 静态库
# ============================================================
add_library(myuart_static STATIC ${SOURCES})
target_include_directories(myuart_static
    PUBLIC
        $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
        $<INSTALL_INTERFACE:include>
    PRIVATE
        ${CMAKE_CURRENT_SOURCE_DIR}/src
)

# ============================================================
# 动态库
# ============================================================
add_library(myuart_shared SHARED ${SOURCES})
target_include_directories(myuart_shared
    PUBLIC
        $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
        $<INSTALL_INTERFACE:include>
    PRIVATE
        ${CMAKE_CURRENT_SOURCE_DIR}/src
)
set_target_properties(myuart_shared PROPERTIES
    VERSION ${PROJECT_VERSION}
    SOVERSION 1
)

# ============================================================
# 安装规则
# ============================================================
install(TARGETS myuart_static myuart_shared
    LIBRARY DESTINATION lib    # .so
    ARCHIVE DESTINATION lib    # .a
)
install(DIRECTORY include/mylib
    DESTINATION include
)
```

## 为什么这样做——逐条解释

### `set(SOURCES src/uart.c src/ringbuffer.c)`

把源文件路径集中在一个变量里，后续两个 `add_library` 复用同一份列表。如果增删源文件，只改这一处。

### 为什么同时建两个目标（static 和 shared）

给使用者选择权：

- 嵌入式 Linux 项目通常倾向**静态链接**——把库代码嵌入固件，避免设备上缺 .so 的风险，也省去运行时加载开销
- 桌面/服务器项目通常倾向**动态链接**——多程序共享、可独立升级 .so

同时产出两种，使用者按需取用。

### `target_include_directories` 中 PUBLIC 和 PRIVATE 的分配

```
PUBLIC:  $<BUILD_INTERFACE:.../include>    → 使用者的编译器能看到 include/mylib/uart.h
PRIVATE: ${CMAKE_CURRENT_SOURCE_DIR}/src   → 只有编译库自己时能找到 uart_internal.h
```

`PRIVATE src/` 保证了内部头文件的隔离——使用者在编译自己的 main.c 时，编译器不会去 src/ 搜头文件，也就找不到 `uart_internal.h`。

### `$<BUILD_INTERFACE:...>` 和 `$<INSTALL_INTERFACE:...>`

这两个生成器表达式处理了编译时和安装后的路径差异：

- 编译库时：头文件在源码树的 `include/` 下 → 用 `$<BUILD_INTERFACE:...>`
- 安装后：头文件被拷贝到 `/usr/local/include/` 下 → 另一个项目 `find_package(myuart)` 时会自动切换到 `$<INSTALL_INTERFACE:include>`

### `VERSION 1.0.0` 和 `SOVERSION 1`

设置动态库的三级命名（详见 [[已有功能打包为库-基础知识]] 第5节 "soname"）：

```
libmyuart.so → libmyuart.so.1 → libmyuart.so.1.0.0
```

### `install(...)`

定义安装目标位置。`LIBRARY DESTINATION lib` 意思是 `.so` 装到 `lib/` 目录（相对于安装前缀 `/usr/local` 就是 `/usr/local/lib/`）。

# 第五步：编译

## 做什么

```bash
cd my_uart
mkdir build && cd build

# 配置（生成 Makefile）
cmake .. -DCMAKE_INSTALL_PREFIX=/usr/local

# 编译
cmake --build .
```

编译完成后，build 目录中的产物：

```
build/
├── libmyuart_static.a        # 静态库
└── libmyuart_shared.so       # 动态库（linker name，实际指向 .so.1.0.0）
    └── libmyuart_shared.so.1.0.0  # 动态库（real name）
```

## 为什么这样做

`mkdir build && cd build` 是"源码外构建"（out-of-source build）——编译产物全部在 build/ 目录中，源码目录保持干净。好处：

- `.gitignore` 只需要忽略 `build/` 一个目录
- 可以同时有多个 build 目录（如 `build_debug`、`build_release`），不同的编译选项互不干扰
- 清理编译产物只需 `rm -rf build`

# 第六步：验证库是否能用

## 做什么

创建 `test/test_uart.c`，写一个最简单的测试：

```c
#include <mylib/uart.h>
#include <stdio.h>

int main() {
    int fd = uart_init("/dev/ttyUSB0", 115200);
    if (fd < 0) {
        printf("无法打开串口（测试环境没有串口也正常）\n");
        return 0;
    }

    const char *msg = "AT\r\n";
    int sent = uart_send(fd, (const uint8_t *)msg, 4);
    printf("发送了 %d 字节\n", sent);

    uart_close(fd);
    return 0;
}
```

然后在 CMakeLists.txt 中追加测试目标（加到 install 规则之前）：

```cmake
# 测试程序——链接动态库来验证 API 正确性
add_executable(test_uart test/test_uart.c)
target_link_libraries(test_uart PRIVATE myuart_shared)
```

重新编译并运行：

```bash
cmake --build .
./test_uart
```

## 为什么这样做

**原因一：验证编译通过。** 测试程序的编译过程验证了"使用者能否正常 include 你的头文件并链接你的库"。这是一个最基础的冒烟测试（smoke test）。

**原因二：验证动态库能加载。** `./test_uart` 运行时如果提示 `cannot open shared object file`，说明动态库的搜索路径有问题——需要设置 LD_LIBRARY_PATH 或 RPATH。这个问题在库的使用者手中出现之前，自己先发现。

**原因三：测试链接的是动态库。** 用 `PRIVATE myuart_shared` 而非 `myuart_static` 来测试，因为动态库的链接和加载问题比静态库更多，先测最复杂的情况。

# 第七步：在其他项目中使用

## 做什么

现在你的库已经编译完成，在另一个项目中使用它：

**方式一：手动指定路径（开发阶段最简单）**

```bash
gcc main.c -I/path/to/my_uart/include -L/path/to/my_uart/build -lmyuart_static -o main
```

**方式二：安装到系统（正式使用）**

```bash
cd my_uart/build
sudo cmake --install .
sudo ldconfig                    # 更新动态链接器缓存
```

安装后系统里的文件分布：

```
/usr/local/
├── include/
│   └── mylib/
│       └── uart.h               # 公开头文件
└── lib/
    ├── libmyuart_static.a       # 静态库
    ├── libmyuart_shared.so → .so.1
    ├── libmyuart_shared.so.1 → .so.1.0.0
    └── libmyuart_shared.so.1.0.0
```

使用者编译时不再需要指定路径：

```bash
gcc main.c -lmyuart_static -o main    # 系统默认搜索 /usr/local/include 和 /usr/local/lib
```

## 为什么这样做

**方式一适合开发期**：库还在频繁修改，不需要反复安装。路径直接指向 build 目录，改了源码重编即可生效。

**方式二适合稳定后**：库接口稳定，安装到标准位置，多个项目可以直接用固定的命令行调用，不用每次指定路径。

# 各步骤之间的承接关系

```
第一步：审视现状
  ↓ 弄清楚输入是什么
第二步：整理目录结构
  ↓ 文件位置定了才能写 CMake
第三步：检查调整头文件
  ↓ 接口搞干净了才不会把内部细节漏出去
第四步：写 CMakeLists.txt
  ↓ 构建脚本是对目录结构和接口的机器可读描述
第五步：编译
  ↓ 产出 .a 和 .so
第六步：写测试验证
  ↓ 确认库能编译通过、能加载
第七步：在其他项目中使用
  ↓ 完成闭环——库被消费者成功链接
```

