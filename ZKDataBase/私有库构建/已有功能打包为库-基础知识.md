# 已有功能打包为库-基础知识

本文档是 [[已有功能打包为库-主文档]] 的配套基础知识文档，按主文档中首次出现的顺序逐一解释涉及的概念和命令。

---

## 1. `set(SOURCES ...)` —— CMake 变量

```cmake
set(SOURCES
    src/uart.c
    src/ringbuffer.c
)
```

`set` 在 CMake 中创建变量，后续用 `${SOURCES}` 引用。这里把源文件列表存在一个变量里，两个 `add_library`（static 和 shared）共用同一份列表。如果增删源文件，只改一行。

---

## 2. `add_library` —— CMake 创建库目标

```cmake
add_library(myuart_static STATIC ${SOURCES})
add_library(myuart_shared SHARED ${SOURCES})
```

**add_library** 向 CMake 注册一个构建目标。

- 第一个参数是目标名称（可以任意取名，建议有意义的名字）
- `STATIC`：产出 `.a`（Linux）或 `.lib`（Windows 静态库）
- `SHARED`：产出 `.so`（Linux）或 `.dll`（Windows）

CMake 自动处理平台差异——`SHARED` 在 Linux 上追加 `-fPIC -shared` 编译选项，在 Windows 上用 `/LD`。

---

## 3. `target_include_directories` 的三级可见性

```cmake
target_include_directories(myuart_shared
    PUBLIC  ...
    PRIVATE ...
)
```

这个命令告诉 CMake："编译 myuart_shared 这个目标时，从哪些目录搜索头文件"。关键在 **PUBLIC / PRIVATE / INTERFACE** 三级控制：

| 关键字 | 库自己编译时能用 | 链接此库的目标能用 | 典型用途 |
|--------|:---:|:---:|---|
| PUBLIC | 是 | 是 | 公开头文件（include/mylib/） |
| PRIVATE | 是 | 否 | 内部头文件（src/） |
| INTERFACE | 否 | 是 | 纯头文件库（只有 .h 没有 .c） |

PRIVATE 的实际效果：

```
编译 myuart_shared 时：   -I./include -I./src     → 能找到 uart.h 和 uart_internal.h
链接 myuart_shared 的 main： -I./include            → 只能找到 uart.h，找不到 uart_internal.h
```

如果 `uart_internal.h` 被标记为 PUBLIC，使用者的代码就可以 `#include "uart_internal.h"` 直接访问内部结构——这是要避免的。

---

## 4. 生成器表达式 `$<BUILD_INTERFACE:...>` `$<INSTALL_INTERFACE:...>`

```cmake
$<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
$<INSTALL_INTERFACE:include>
```

**为什么需要两个路径？**

编译库时和安装后的物理路径不同：

| 阶段 | 头文件在哪里 | include 路径 |
|------|------------|-------------|
| 编译库时 | 源码树中 | `/home/user/my_uart/include` |
| 安装后，其他项目用它 | 系统目录中 | `/usr/local/include` |

`$<BUILD_INTERFACE:...>` 在编译库时生效，指向源码树的真实路径。

`$<INSTALL_INTERFACE:include>` 在安装后生效——另一个项目 `find_package(myuart)` 时，CMake 自动把 `include` 拼上安装前缀（`/usr/local/`），变成 `/usr/local/include`。

这是 CMake 的**生成器表达式**——它不是 CMake 变量，表达式在 Makefile 生成阶段（而非配置阶段）才求值。

## 5. soname 与动态库三级命名

`set_target_properties` 中设置的 `VERSION` 和 `SOVERSION` 决定了动态库文件名：

```cmake
set_target_properties(myuart_shared PROPERTIES
    VERSION ${PROJECT_VERSION}    # 1.0.0
    SOVERSION 1                   # 1
)
```

产生三个文件：

```
libmyuart_shared.so        → 符号链接 → libmyuart_shared.so.1
libmyuart_shared.so.1      → 符号链接 → libmyuart_shared.so.1.0.0
libmyuart_shared.so.1.0.0  → 真实文件
```

| 名称 | 谁在用 | 何时用 |
|------|--------|--------|
| `libmyuart_shared.so` | 编译器（linker name） | 编译时 `-lmyuart_shared` 找到这个链接 |
| `libmyuart_shared.so.1` | 动态链接器（soname） | 运行时按 soname 找文件 |
| `libmyuart_shared.so.1.0.0` | 文件系统（real name） | 实际文件，存储字节数据 |

**maojor版本号变动的规则**：`SOVERSION` 是主版本号。接口不兼容（删了函数、改了函数签名）时 `SOVERSION` 要 +1，旧程序就无法加载新 .so。新增函数不改 `SOVERSION`，旧程序不受影响。

## 6. cmake 的 out-of-source build（源码外构建）

标准做法是建一个独立的 build 目录，在 build 里执行 cmake：

```bash
mkdir build && cd build
cmake ..
cmake --build .
```

**为什么不在源码目录直接 cmake？**

直接 `cmake .` 会原地生成 Makefile、CMakeCache.txt、编译产物（.o、.a、.so），混在源码目录里：

```
my_uart/
├── CMakeCache.txt        ← 污染
├── Makefile              ← 污染
├── CMakeFiles/           ← 污染
├── src/uart.c.o          ← 污染
├── libmyuart.a           ← 污染
├── src/
│   └── uart.c            ← 源码
└── include/
    └── ...
```

源码外构建把所有中间产物隔离在 build/ 目录中，好处：
- `.gitignore` 只需要写 `build/` 一行
- `rm -rf build` 一键清理
- 可以并存多个构建配置：`build_debug/`、`build_release/`

## 7. `cmake --build` 和 `cmake --install`

```bash
cmake --build .           # 执行编译（调用 make / ninja / msbuild）
cmake --install .         # 执行安装（拷贝文件到目标目录）
```

**`cmake --build .`**：CMake 不知道底层用什么构建工具，`--build` 让它自己去调用正确的后端——Linux 上通常是 make，Windows 上可能是 MSBuild。这比直接敲 `make` 更通用。

**`cmake --install .`**：按 CMakeLists.txt 中 `install()` 指令的定义，把文件拷贝到 `${CMAKE_INSTALL_PREFIX}` 指定的位置。

```
install(TARGETS myuart_static myuart_shared
    LIBRARY DESTINATION lib      # .so → ${CMAKE_INSTALL_PREFIX}/lib/
    ARCHIVE DESTINATION lib      # .a  → ${CMAKE_INSTALL_PREFIX}/lib/
)
install(DIRECTORY include/mylib
    DESTINATION include          # 头文件 → ${CMAKE_INSTALL_PREFIX}/include/mylib/
)
```

## 8. `target_link_libraries` 中的 PRIVATE

```cmake
target_link_libraries(test_uart PRIVATE myuart_shared)
```

PRIVATE 的含义：test_uart 使用 myuart_shared，但 test_uart 的使用者不会获得 myuart_shared。

如果用 PUBLIC：
```cmake
target_link_libraries(test_uart PUBLIC myuart_shared)
# 意味着"任何人链接 test_uart，就自动链接 myuart_shared"
```
test_uart 是一个测试程序，不会有人链接它，所以用 PRIVATE 就够了。

PUBLIC 的正确使用场景：库 A 的公开头文件中包含了库 B 的类型，那么库 A 应该 `target_link_libraries(A PUBLIC B)`。

## 9. `LD_LIBRARY_PATH` 环境变量

动态链接器在搜索 .so 时的查找顺序之一（排在 RPATH/RUNPATH 之后、ldconfig 缓存之前）：

```bash
LD_LIBRARY_PATH=/path/to/lib ./test_uart
```

- 用于**临时测试**——开发阶段不想每次装到系统路径，用这个变量直接指向 build/ 目录
- 不适合生产部署——不应该要求用户设置环境变量才能运行程序
- 正式的解决方案是 RPATH（编译时写入可执行文件）或者安装到系统路径

---

## 10. ldconfig

更新 Linux 动态链接器（ld-linux.so）的缓存：

```bash
sudo ldconfig            # 扫描 /lib、/usr/lib、/etc/ld.so.conf 中的路径
sudo ldconfig -p         # 列出缓存中所有已知的 .so
```

安装新的 .so 到 `/usr/local/lib` 后，必须执行 `sudo ldconfig`。否则动态链接器不知道它在那里——即使文件在标准路径下。

不执行 ldconfig 的后果：

```bash
sudo cmake --install .          # .so 已经拷贝到 /usr/local/lib/
./test_uart                     # 报错：cannot open shared object file
sudo ldconfig                   # 更新缓存
./test_uart                     # 正常运行
```

## 11. `#include` 的双引号 vs 尖括号

```c
#include "uart.h"              // 双引号：先在当前目录搜，再搜 -I 路径
#include <mylib/uart.h>        // 尖括号：只搜 -I 路径和系统路径
```

库的头文件用尖括号引用——表明"这个头文件不在当前目录，在系统或指定的 include 路径中"。这是 C 语言的标准约定，向读代码的人传递"这是外部库的头文件，不是本项目的文件"。

## 12. 头文件保护宏的命名

```c
#ifndef MYLIB_UART_H      // 防止重复包含
#define MYLIB_UART_H

// ... 声明 ...

#endif
```

宏名加入库名前缀（`MYLIB_`）是为了避免和其他库的保护宏冲突。如果两个库的 uart.h 都写 `#ifndef UART_H`，后包含的会被跳过（因为第一次 `#include` 时已经定义了 `UART_H`）。
