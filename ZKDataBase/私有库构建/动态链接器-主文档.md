# 动态链接器-主文档

## 本文主线

动态链接器是什么、在程序启动的哪一步介入工作、按什么顺序搜索 .so、有哪些常见问题需要注意。

---

# 第一章：它是什么

**动态链接器（ld-linux.so）是操作系统加载动态链接的可执行文件时，最先运行的一段代码。** 它不是编译器（gcc），也不是静态链接器（ld），它是运行时组件。

物理位置：

```bash
ls -l /lib64/ld-linux-x86-64.so.2      # x86-64
ls -l /lib/ld-linux-armhf.so.3         # ARM 32位
```

每个动态链接的可执行文件的 ELF 头中写死了谁来加载它：

```bash
readelf -l ./main | grep interpreter
# [Requesting program interpreter: /lib64/ld-linux-x86-64.so.2]
```

`PT_INTERP` 段里存的这个路径就是动态链接器。内核加载可执行文件时读到它，先加载动态链接器，再把控制权交给动态链接器。

---

# 第二章：它在哪一步工作——完整启动链路

```
1. 终端敲 ./main
       │
2. shell 调用 execve() 系统调用
       │
3. 内核读取 main 的 ELF 头
       │
4. 内核发现 .interp 段写着需要 /lib64/ld-linux-x86-64.so.2
       │
5. 内核加载 ld-linux-x86-64.so.2 到内存
       │
6. 内核把控制权交给动态链接器  ★← 动态链接器在此刻接手
       │
7. 动态链接器读取 main 的 .dynamic 段，找到依赖列表：
   libc.so.6, libmyuart.so, libpthread.so.0
       │
8. 动态链接器按搜索优先级逐个找到这些 .so，加载进内存
       │
9. 动态链接器做重定位——修正 GOT 和 PLT 中的函数地址
       │
10. 动态链接器调用各 .so 的初始化函数
       │
11. 动态链接器跳转到 main 的 _start → main()
       │
12. 你的代码开始执行
```

动态链接器在第 6 步接手，第 11 步交棒。它做的事情对程序透明——程序里写 `printf("hello")` 时不需要知道 libc.so 在哪，动态链接器已经找好并修正了 printf 的地址。

# 第三章：搜索 .so 的优先级

动态链接器的搜索有严格顺序，硬编码在源码中：

```
① RPATH      —— 编译时通过 -Wl,-rpath 写入 ELF 的路径
② RUNPATH    —— 编译时写入的新式路径
③ LD_LIBRARY_PATH —— 环境变量
④ /etc/ld.so.cache —— ldconfig 生成的索引缓存
⑤ /lib → /usr/lib → /usr/local/lib —— 系统默认路径
⑥ 报错：cannot open shared object file
```

每一项的优先级是动态链接器代码决定的，不能修改。能控制的是前三项：RPATH/RUNPATH（编译时决定）、LD_LIBRARY_PATH（运行时决定）。

# 第四章：需要注意的问题

## 4.1 RPATH vs RUNPATH——LD_LIBRARY_PATH 能否覆盖？

- **RPATH**（老式）：优先级**高于** LD_LIBRARY_PATH → 环境变量无法覆盖
- **RUNPATH**（新式）：优先级**低于** LD_LIBRARY_PATH → 环境变量可以覆盖

CMake 默认用 RUNPATH。如果程序是别人编译的、RPATH 中写死了一个不存在的路径，LD_LIBRARY_PATH 也救不回来。

查看当前程序用的是哪个：

```bash
readelf -d ./main | grep -E "RPATH|RUNPATH"
# 0x000000000000001d (RUNPATH)  Library runpath: [/usr/local/lib]
```

## 4.2 LD_LIBRARY_PATH 加载到错误的 .so

LD_LIBRARY_PATH 在系统路径和 ldconfig 缓存之前被搜索。如果把有旧版系统库的目录加进去，可能导致所有命令崩溃。

安全做法——永远追加而非覆盖：

```bash
# 错误
export LD_LIBRARY_PATH=/home/user/mylibs

# 正确
export LD_LIBRARY_PATH=$LD_LIBRARY_PATH:/home/user/mylibs
```

## 4.3 架构不匹配

x86-64 编译的 .so 不能在 ARM 上加载。动态链接器检查 ELF 头的架构字段，不匹配直接拒绝：

```bash
./main
# wrong ELF class: ELFCLASS64 (or ELFCLASS32)
```

排查：

```bash
file libmyuart.so
# libmyuart.so: ELF 64-bit LSB shared object, x86-64  ← 和目标的架构比对
```

## 4.4 soname 主版本号不匹配

编译时链接的 soname 是 `libmyuart.so.1`，运行时只有 `libmyuart.so.2`。动态链接器比较 soname，主版本号不同就拒绝加载：

```bash
./main
# libmyuart.so.1: cannot open shared object file
#（实际上 .so.2 存在，但 soname 不匹配）
```

修复方式二选一：
- 重新编译程序链接新版库
- 保留旧版本 .so 并存（两个主版本号的文件同时存在）

## 4.5 GOT 重定位失败——符号找不到

动态链接器加载了 .so，但 .so 中引用另一个 .so 的符号不存在：

```bash
./main
# symbol lookup error: /usr/lib/libmyuart.so: undefined symbol: some_func
```

用 ldd 递归检查依赖链：

```bash
ldd libmyuart.so       # 看所有依赖是否齐全
ldd -r libmyuart.so    # 同时报告未解析符号
```
