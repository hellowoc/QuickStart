# 环境变量配置-主文档

## 本文主线

环境变量是什么、做库/用库的流程中涉及哪些环境变量、各在什么环节配置、有什么坑。

---

# 第一章：环境变量是什么

## 1.1 数据结构视角

**环境变量是操作系统为每个进程维护的一组 key=value 键值对。** 它不是存在文件里的配置，而是进程内存中的一块数据。每个进程启动时，操作系统从父进程拷贝一份环境变量表给它。

```c
// 进程内存中的环境变量表（简化）
char *environ[] = {
    "HOME=/home/user",
    "PATH=/usr/bin:/bin:/usr/local/bin",
    "LD_LIBRARY_PATH=/opt/myapp/lib",
    "USER=zhang",
    NULL           // 数组结束标志
};
```

程序用 `getenv()` 读取、`setenv()` 写入：

```c
#include <stdlib.h>

const char *home = getenv("HOME");     // 读：返回 "/home/user"
setenv("MY_VAR", "hello", 1);          // 写：设 MY_VAR=hello
```

## 1.2 隔离性

进程只能修改**自己**的环境变量，不影响父进程或其他进程。在当前终端执行 `export FOO=bar`，这个终端启动的所有子进程都能看到 `FOO=bar`，但另一个终端窗口看不到。

# 第二章：库相关的三个环境变量

| 环境变量 | 谁读取它 | 什么时候读取 | 作用 |
|----------|---------|------------|------|
| `LD_LIBRARY_PATH` | 动态链接器 (ld-linux.so) | 运行时 | 告诉链接器去哪些路径找 .so |
| `PATH` | shell | 敲命令时 | 找到可执行文件（如 cmake、gcc） |
| `CMAKE_PREFIX_PATH` | CMake | cmake 配置时 | 找 `find_package` 的库 |

---

# 第三章：LD_LIBRARY_PATH —— 核心变量

## 3.1 使用场景

库编译出了 `libmyuart.so`，放在 `/home/user/my_uart/build/`。程序 `test_uart` 链接了这个库，运行时动态链接器找不到：

```bash
./test_uart
# error while loading shared libraries: libmyuart.so: cannot open shared object file
```

设置 `LD_LIBRARY_PATH` 后正常运行：

```bash
LD_LIBRARY_PATH=/home/user/my_uart/build ./test_uart
```

## 3.2 三种配置方式

**方式一：单次生效**

```bash
LD_LIBRARY_PATH=/path/to/lib ./your_program
```

变量设置在命令前，只影响这一次执行。

**方式二：当前终端会话生效**

```bash
export LD_LIBRARY_PATH=/path/to/lib:$LD_LIBRARY_PATH
./your_program
```

`$LD_LIBRARY_PATH` 把新路径追加到已有值后面。`export` 让子进程也能读到。

**方式三：永久生效（写入 ~/.bashrc）**

```bash
echo 'export LD_LIBRARY_PATH=/home/user/my_libs:$LD_LIBRARY_PATH' >> ~/.bashrc
source ~/.bashrc
```

适合库已稳定、放在固定位置长期使用的情况。

## 3.3 安全注意

LD_LIBRARY_PATH 优先级高于系统路径。如果把它指向了有旧版系统库的目录，可能导致所有命令崩溃。

```bash
# 错误：不加 :$LD_LIBRARY_PATH，覆盖了原有值
export LD_LIBRARY_PATH=/home/user/mylibs

# 正确：追加
export LD_LIBRARY_PATH=$LD_LIBRARY_PATH:/home/user/mylibs
```

# 第四章：PATH —— 可执行文件的搜索路径

shell 接收命令后，去 PATH 列出的目录逐个搜索同名可执行文件：

```bash
echo $PATH
# /usr/local/bin:/usr/bin:/bin

# 追加自定义工具目录
export PATH=$HOME/tools/bin:$PATH
```

做库过程中一般不需要改 PATH，但如果交叉编译器（arm-linux-gcc）不在 PATH 里会报 "command not found"。

---

# 第五章：CMAKE_PREFIX_PATH —— 让 CMake 找到非系统库

库 A 依赖库 B，库 B 是自己编译的、不在系统标准路径，需要告诉 CMake 去哪找：

```bash
cmake .. -DCMAKE_PREFIX_PATH=/home/user/my_libs
```

或者在 CMakeLists.txt 中直接写（推荐，不依赖环境变量）：

```cmake
find_package(mylib REQUIRED PATHS /home/user/my_libs)
```

# 第六章：全流程总结

```
写代码阶段    → 不需要任何环境变量
cmake 配置    → CMAKE_PREFIX_PATH（库依赖了其他非系统库时）
cmake --build → PATH（确保 cmake、gcc 可被找到）
运行测试      → LD_LIBRARY_PATH（让动态链接器找到你的 .so）
安装到系统    → 不再需要 LD_LIBRARY_PATH（ldconfig 接管）
分发给别人    → 对方需要 LD_LIBRARY_PATH 或 RPATH
```

最常用的开发期命令：

```bash
LD_LIBRARY_PATH=./build ./build/test_uart
```

---

# 第七章：不依赖 LD_LIBRARY_PATH 的方案：RPATH

编译时把搜索路径写进可执行文件的 ELF 头：

```bash
gcc main.o -L./build -lmyuart -Wl,-rpath,/home/user/my_uart/build -o main
```

CMake 方式：

```cmake
target_link_options(test_uart PRIVATE "-Wl,-rpath,${CMAKE_BINARY_DIR}")
```

运行时动态链接器优先读 RPATH，不需要 LD_LIBRARY_PATH。
