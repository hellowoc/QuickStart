# 私有库构建-基础知识

本文档是 [[私有库构建-主文档]] 的配套基础知识文档。按主文档中首次出现的顺序，逐一解释涉及的基础概念、工具和命令。

---

## 1. 目标文件 .o

**.o 文件**是 C/C++ 编译器输出的中间产物。一个 .c 文件经过预处理和编译后生成一个 .o 文件。.o 文件不是纯二进制机器码，它遵循目标文件格式规范（Linux 上是 ELF 格式，Windows 上是 COFF/PE 格式）。

.o 文件的核心结构（ELF 格式）：

```
Header          —— 文件类型（ET_REL 表示可重定位文件）、目标架构
.text           —— 代码段：函数编译后的机器指令
.data           —— 已初始化全局变量（int x = 5）
.rodata         —— 只读数据：字符串常量、const 变量
.bss            —— 未初始化全局变量（int x;），不占实际磁盘空间，只记录大小
.symtab         —— 符号表：本文件定义的符号 + 引用的外部符号
.rela.text      —— 代码段重定位表：哪些指令中的地址需要修正
.rela.data      —— 数据段重定位表：哪些数据指针需要修正
.strtab         —— 字符串表：符号名等字符串
```

查看 .o 文件的结构：

```bash
file hello.o                    # 确认文件类型
readelf -S hello.o              # 列出所有段
readelf -s hello.o              # 查看符号表
objdump -d hello.o              # 反汇编代码段
nm hello.o                      # 快速查看符号
```

## 2. 段（Section）

**段**是目标文件中的逻辑分区。每个段存储一类数据：

| 段名 | 内容 | 权限 |
|------|------|------|
| .text | 可执行机器指令 | r-x（可读可执行） |
| .data | 已初始化的全局变量和静态变量 | rw-（可读可写） |
| .rodata | 只读数据（字符串常量、const 变量） | r--（只读） |
| .bss | 未初始化（或初始化为0）的全局变量 | rw- |

.bss 段不占用实际磁盘空间——它在文件头中记录"加载时需要 N 字节零页"，由操作系统在加载时分配并清零。

## 3. 符号与符号表 .symtab

**符号**是代码中函数名、全局变量名和静态变量名的统称。符号表是 .o/.so/可执行文件中记录所有符号的结构。

符号的核心属性：

```c
// 符号表中的一个条目（简化）
struct symtab_entry {
    char *name;        // 符号名（如 "hello_say"）
    void *address;     // 符号在段内的偏移（.o 中）或虚拟地址（.so/可执行文件中）
    int binding;       // LOCAL（static函数，外部不可见）/ GLOBAL（外部可见）/ WEAK
    int type;          // FUNC（函数）/ OBJECT（变量）/ NOTYPE
    int section_index; // 符号属于哪个段（.text / .data / UNDEF 表示未定义，需要外部提供）
};
```

`nm` 命令输出的列含义：

```bash
nm hello.o
# 0000000000000000 T hello_say      ← T = .text 中的全局（导出）函数
# 0000000000000000 D version_num    ← D = .data 中的全局变量
# 0000000000000000 R version_str    ← R = .rodata 中的只读数据
#                  U printf          ← U = 未定义，需要外部提供（在 libc.so 中）
# 0000000000000000 t internal_static ← t = .text 中的 LOCAL（static）函数
```

## 4. 重定位表 .rela.text / .rela.data

编译一个 .c 文件时，编译器不知道外部函数（如 printf）在内存中的最终地址。程序启动后的地址空间布局是链接器和动态链接器决定的。

**重定位表**记录了"哪些位置的地址需要修正"以及"修正的依据是什么符号"：

```c
// 重定位条目的核心字段
struct relocation_entry {
    void *offset;       // 在 .text 或 .data 中需要修正的位置（相对于段起始）
    int type;           // 重定位类型（R_X86_64_PC32 相对跳转 / R_X86_64_64 绝对地址）
    int symbol_index;   // 这个位置引用了哪个符号（索引到 .symtab）
    int64_t addend;     // 计算最终地址时的附加常数
};
```

静态链接时，链接器根据重定位表把所有地址修正为最终地址。动态链接时，动态链接器在加载 .so 时完成修正。

## 5. ar 工具

**ar**（archiver）用于创建、修改、提取静态库（.a 文件）。

常用选项：

```bash
ar rcs libhello.a hello.o world.o  # r=插入, c=创建, s=写入索引
ar t libhello.a                     # 列出 .a 中包含的文件
ar x libhello.a                     # 提取 .a 中的全部文件
ar d libhello.a hello.o             # 从 .a 中删除指定文件
```

`r`（replace）、`c`（create）、`s`（symbol index）三个选项几乎总是连用。

`s` 是 ar 的关键操作——它创建一个符号索引表（`__.SYMDEF`），记录"哪个符号在哪个 .o 文件中"。链接器用这个索引快速定位需要的 .o，而不需要扫描 .a 中的每一个 .o 的符号表。

## 6. ELF 文件格式

**ELF**（Executable and Linkable Format）是 Linux 上目标文件、动态库、可执行文件的统一文件格式。

三种 ELF 类型：

| 类型 | file 命令输出 | ET_REL / ET_EXEC / ET_DYN |
|------|--------------|--------------------------|
| .o 目标文件 | `ELF 64-bit LSB relocatable` | ET_REL |
| .so 动态库 | `ELF 64-bit LSB shared object` | ET_DYN |
| 可执行文件 | `ELF 64-bit LSB executable` | ET_EXEC |

动态库和可执行文件的结构相似——都有完整的 Program Header Table（告诉内核如何映射到内存）、Section Header Table、符号表、重定位表等。

查看 ELF 结构的关键命令：

```bash
file <file>          # 判断文件类型
readelf -h <file>    # ELF 头（类型、入口点、段表/节表位置）
readelf -S <file>    # 所有 Section（节）
readelf -l <file>    # 所有 Segment（加载段）
readelf -d <file>    # .dynamic 段（动态链接信息：soname、依赖、RPATH）
readelf -s <file>    # 符号表
```

## 7. PE 文件格式

Windows 上的可执行文件格式，.exe 和 .dll 都是 PE 格式。与 ELF 的函数对应：

| 概念 | ELF (Linux) | PE (Windows) |
|------|------------|--------------|
| 目标文件后缀 | .o | .obj |
| 静态库 | .a | .lib |
| 动态库 | .so | .dll |
| 导入库（链接 .dll 时需要） | 不需要 | .lib（与静态库同名但内容不同） |
| 符号导出标记 | `visibility("default")` | `__declspec(dllexport)` |
| 符号导入标记 | 不需要 | `__declspec(dllimport)` |
| 运行时搜索 | RPATH/RUNPATH/LD_LIBRARY_PATH | PATH + 同目录 |

## 8. GOT 与 PLT

这两个概念是 `-fPIC` 的基础。

**GOT**（Global Offset Table，全局偏移表）：.so 中的一个指针数组，每个条目指向一个全局变量或函数的实际地址。编译时不知道符号地址，所以在 GOT 中分配一个槽位，运行时动态链接器填充正确的地址值。代码通过 GOT 间接访问，而不是直接用绝对地址。

**PLT**（Procedure Linkage Table，过程链接表）：.so 中的一段跳板代码，用于实现函数的延迟绑定（lazy binding）。外部函数首次调用时才解析地址，解析后后续调用直接跳转。

延迟绑定的流程：
1. 代码调用 `printf@PLT`
2. PLT 跳转到 GOT 中 printf 对应的条目
3. GOT 条目初始指向动态链接器的解析函数
4. 解析函数查找 printf 的真实地址，更新 GOT 条目
5. 下一次调用 `printf@PLT` → 跳转到 GOT → 直接到 printf（不再经过解析函数）

`-fPIC` 生成的代码通过 GOT 访问全局数据、通过 PLT 调用外部函数。这样无论 .so 被加载到哪个虚拟地址，代码段指令不变，只需修改 GOT 中少量的指针。

## 9. 动态链接器 ld-linux.so

ld-linux.so 是 ELF 可执行文件指定的程序解释器（program interpreter），负责：
1. 加载可执行文件依赖的所有 .so
2. 执行重定位：修正 GOT 中的地址
3. 调用各 .so 的构造函数（`__attribute__((constructor))`）
4. 最后跳转到可执行文件的入口点（_start → main）

查看可执行文件指定的解释器：

```bash
readelf -l ./main | grep interpreter
# [Requesting program interpreter: /lib64/ld-linux-x86-64.so.2]
```

## 10. 编译器与链接器关键选项

### gcc -c
编译并汇编，产出 .o 文件，不进行链接。

```bash
gcc -c hello.c -o hello.o
```

### gcc -fPIC
生成位置无关代码（Position Independent Code）。编译产出通过 GOT/PLT 间接访问符号，使 .o 可以被链接进 .so 并在任意内存地址加载。

### gcc -shared
告诉链接器产出动态库而非可执行文件。链接器不会报"缺少 main 函数"。

### gcc -L
指定链接器搜索库文件的目录。

```bash
gcc main.o -L./mylibs -lhello -o main
# 链接器在 ./mylibs/ 中搜索 libhello.so 或 libhello.a
```

### gcc -l
指定要链接的库名。去掉 `lib` 前缀和 `.so`/`.a` 后缀。

```bash
-lhello          # 搜索 libhello.so 或 libhello.a
-l:libhello.a    # 精确指定文件名
```

### gcc -I
指定预处理器搜索头文件的目录。

```bash
gcc -c main.c -I./include -o main.o
```

### gcc -Wl,-rpath,\<path\>
向链接器传递 `-rpath <path>` 选项，将路径写入可执行文件的 RPATH 字段。运行时动态链接器优先从这个路径搜索 .so。

`-Wl,` 是 gcc 的前缀，意思是"把后面的内容传给链接器 ld"。

## 11. LD_LIBRARY_PATH

环境变量。运行时动态链接器在 RPATH/RUNPATH 之后、ldconfig 缓存之前搜索此变量中的路径。

```bash
LD_LIBRARY_PATH=/opt/custom/lib ./myapp
```

对临时测试和开发环境有用。不适合生产部署——不应该要求用户设置环境变量才能运行程序。

---

## 12. soname 机制

soname（Shared Object Name）是嵌入在 .so 的 ELF 头的 `.dynamic` 段中的一个字符串，格式为 `lib<N>.so.<MAJOR>`。

```bash
# 编译时给 .so 设置 soname
gcc -shared -Wl,-soname,libhello.so.1 -o libhello.so.1.0.0 hello.o

# 查看 .so 的 soname
readelf -d libhello.so.1.0.0 | grep SONAME
# 0x000000000000000e (SONAME)  Library soname: [libhello.so.1]
```

链接器链接可执行文件时，把 soname（不是 real name 文件名）记录到可执行文件的依赖列表中。运行时动态链接器按 soname 去搜索，找到 `libhello.so.1`（符号链接） → `libhello.so.1.0.0`（真实文件）。

这个机制的意义：升级到 libhello.so.1.0.1 时只需更新符号链接指向新文件，依赖的可执行文件不用重新链接。

## 13. RPATH 与 RUNPATH

两者都存储在 ELF 头的 `.dynamic` 段中，都是告诉动态链接器优先搜索哪些路径。

区别在于搜索顺序：

- **RPATH**（旧）：在 LD_LIBRARY_PATH **之前**搜索。不可被环境变量覆盖
- **RUNPATH**（新）：在 LD_LIBRARY_PATH **之后**搜索。允许环境变量覆盖

CMake 默认使用 RUNPATH（通过 `--enable-new-dtags` 链接选项）。

两者的共同替代方案是 `$ORIGIN`——在路径中写 `$ORIGIN/../lib`，动态链接器把 `$ORIGIN` 展开为可执行文件所在目录。

## 14. nm 命令

列出目标文件/库/可执行文件中的符号。

```bash
nm hello.o          # 查看 .o 的符号
nm -D libhello.so   # 查看 .so 的动态符号表（可被外部链接的符号）
nm libhello.a       # 查看 .a 中每个 .o 的符号
```

输出格式：`<地址> <类型> <符号名>`

| 类型字母 | 含义 |
|---------|------|
| T / t | 代码段中的函数（大写=GLOBAL, 小写=LOCAL） |
| D / d | .data 段中的已初始化变量 |
| B / b | .bss 段中的未初始化变量 |
| R / r | .rodata 段中的只读数据 |
| U | 未定义符号（需要外部提供） |
| W | 弱符号（可被同名的强符号覆盖） |

## 15. ldd 命令

列出可执行文件或 .so 依赖的所有动态库，以及每个库的解析路径。

```bash
ldd ./main
# linux-vdso.so.1 =>  (0x00007ffe...)
# libhello.so => /usr/local/lib/libhello.so (0x00007f...)
# libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6 (0x00007f...)
# /lib64/ld-linux-x86-64.so.2 (0x00007f...)
```

某行显示 `not found` 时，说明动态链接器无法找到该库，程序将无法运行。

## 16. readelf 命令

详细查看 ELF 文件的内部结构。

```bash
readelf -h <file>     # ELF 文件头
readelf -l <file>     # Program Headers（加载段，内核用）
readelf -S <file>     # Section Headers（节，链接器用）
readelf -d <file>     # .dynamic 段（依赖、soname、RPATH）
readelf -s <file>     # 符号表
readelf -r <file>     # 重定位表
```

判断文件类型——基于文件头魔数（magic number）而非扩展名。

## 18. GCC 符号可见性属性

### __attribute__((visibility("default")))

函数/变量用此属性标记后，它在 .so 的动态符号表中出现，外部可链接。

```c
__attribute__((visibility("default")))
void public_func(void) { ... }
```

### __attribute__((visibility("hidden")))

函数/变量用此属性标记后，它不出现在动态符号表中，仅库内部可用。

### -fvisibility=hidden

编译选项，把**所有**非显式标记的符号默认设为 hidden。加上此选项后，只有显式标记了 `visibility("default")` 的符号才会被导出。

```bash
gcc -c -fPIC -fvisibility=hidden api.c -o api.o
```

## 19. Windows 符号导出宏

### __declspec(dllexport)

编译 .dll 时，用此属性标记要导出的函数/变量。

```c
__declspec(dllexport)
void public_func(void) { ... }
```

### __declspec(dllimport)

使用 .dll 时，用此属性声明导入的函数/变量。告诉编译器"这个符号在 .dll 中，不要在本模块中找定义"。

```c
__declspec(dllimport)
void public_func(void);
```

Linux 不需要 dllimport——.so 的符号全部在编译时通过 PLT 间接访问，编译器不需要额外的导入声明。

## 20. CMake 构建库的核心命令

### add_library

```cmake
add_library(<target> STATIC <sources>)    # 产出 .a（Linux） 或 .lib（Windows静态库）
add_library(<target> SHARED <sources>)    # 产出 .so（Linux） 或 .dll（Windows）
add_library(<target> OBJECT <sources>)    # 只编译到 .o，不打包/链接（中间形态）
```

CMake 自动处理平台差异：SHARED 在 Linux 上加 `-fPIC -shared`，在 Windows 上设置 `/LD`。

### target_include_directories

```cmake
target_include_directories(<target>
    PUBLIC <dirs>      # 自己 + 链接此目标的目标都能看到这些头文件
    PRIVATE <dirs>     # 只有自己能看到
    INTERFACE <dirs>   # 自己不看到，但链接此目标的目标能看到
)
```

PUBLIC 对应库的公开头文件（include/），PRIVATE 对应内部头文件（src/），INTERFACE 用于纯头文件库（header-only library）。

### target_link_libraries 的 PUBLIC/PRIVATE

```cmake
target_link_libraries(mylib PRIVATE pthread)
# mylib 使用 pthread，但不对外暴露
# 链接 mylib 的目标不会自动链接 pthread

target_link_libraries(mylib PUBLIC openssl)
# mylib 公开 API 包含 OpenSSL 类型（如头文件中有 #include <openssl/ssl.h>）
# 链接 mylib 的目标自动获得 OpenSSL 的 include 路径和链接
```

### target_compile_definitions

```cmake
target_compile_definitions(mylib PRIVATE MYLIB_BUILDING)
# 编译 mylib 时定义 MYLIB_BUILDING 宏
# 等价于 gcc -DMYLIB_BUILDING
```

### install

```cmake
install(TARGETS mylib_shared mylib_static
    LIBRARY DESTINATION lib    # SHARED 库
    ARCHIVE DESTINATION lib    # STATIC 库
)
install(DIRECTORY include/mylib
    DESTINATION include
)
```

### 生成器表达式 `$<...>`

```cmmake
$<BUILD_INTERFACE:/path/to/include>     # 编译时使用此路径
$<INSTALL_INTERFACE:include>            # 安装后使用此路径
```

编译 mylib 时，include 路径指向源码树中的 `include/`。安装后另一个项目用 `find_package` 链接 mylib 时，路径自动切换为 `/usr/local/include`。

## 21. GenerateExportHeader

CMake 模块，自动生成跨平台的导出宏头文件。

```cmake
include(GenerateExportHeader)
generate_export_header(mylib_shared
    BASE_NAME mylib
    EXPORT_FILE_NAME include/mylib/mylib_export.h
)
```

生成的宏根据平台和编译上下文自动切换：
- 编译库时（CMake 定义了 `mylib_shared_EXPORTS`）→ 展开为 `__declspec(dllexport)`（Win）或 `visibility("default")`（Linux）
- 使用库时（`mylib_shared_EXPORTS` 未定义）→ 展开为 `__declspec(dllimport)`（Win）或空（Linux）

## 22. ldconfig

Linux 动态链接器缓存管理工具。

```bash
sudo ldconfig                    # 扫描 /lib, /usr/lib, /etc/ld.so.conf 中列出的路径
                                 # 把找到的 .so 的 soname → 文件路径映射写入 /etc/ld.so.cache
sudo ldconfig -p                 # 打印当前缓存中的所有库
sudo ldconfig -p | grep hello    # 查找特定库是否在缓存中
```

安装新的 .so 到系统路径后，必须运行 `ldconfig` 更新缓存，否则动态链接器找不到它。

## 23. $ORIGIN

动态链接器识别的特殊变量，展开为可执行文件所在的目录的绝对路径。

```bash
# 编译时设置 RPATH 为 $ORIGIN/../lib
gcc main.o -L. -lhello -Wl,-rpath,'$ORIGIN/../lib' -o main

# 运行时，无论程序安装在哪个目录：
#   /opt/myapp/bin/main  →  $ORIGIN = /opt/myapp/bin  →  搜索 /opt/myapp/lib
#   /home/user/bin/main  →  $ORIGIN = /home/user/bin  →  搜索 /home/user/lib
```

注意 shell 中转义——单引号防止 shell 展开 `$ORIGIN`，把它原样传给链接器。
