# 私有库构建-主文档

## 本文主线

从零开始构建一个私有 C/C++ 库。起点是"库到底是什么"——用 gcc 手写命令看清 .o → .a → .so 的完整链路；然后进入工程化：目录结构 → CMake 构建脚本 → 符号导出控制 → 安装与运行时查找 → 跨平台适配；终点是能够用一套 CMakeLists.txt 同时产出静态库和动态库，并以 mosquitto 的真实构建脚本作为参考对照。

全流程覆盖：源码 → 编译 → 打包 → 安装 → 被其他程序链接使用。

---

# 第一章：库是什么——.o 文件的两种组织方式

## 1.1 库的本质

一个 C/C++ 程序从源码到可执行文件，中间产物是 **目标文件**（.o）。每个 .c 文件编译后生成一个 .o 文件，.o 文件包含：

- **代码段**（.text）：编译后的机器指令
- **数据段**（.data / .rodata / .bss）：全局变量、静态变量、字符串常量
- **符号表**（.symtab）：这个 .o 提供了哪些函数/变量（导出符号），需要哪些外部函数/变量（未定义符号）
- **重定位表**（.rela.text / .rela.data）：哪些地址需要在链接时修正

"库"就是把多个 .o 文件组织成一个文件，然后告诉链接器"这里的符号你都可以用"。根据组织方式的不同，分为静态库和动态库。

## 1.2 静态库（.a / .lib）

**本质**：把多个 .o 文件用 `ar` 工具打成一个归档包。ar = archive。

```
libhello.a = hello.o + world.o + utils.o  （ar 打包）
```

可以理解为：把几个 .o 文件用 tar 打了个包，然后链接时从包里取出需要的 .o。

**链接行为**：链接器（ld）在链接可执行文件时，从 .a 中找到需要的 .o，把 .o 的代码段和数据段**直接拷贝**进最终的可执行文件。

**结果**：
- 可执行文件包含所有用到的库代码 → 文件体积大
- 运行时不需要 .a 文件 → 部署简单，只有一个可执行文件
- 如果 10 个程序都用同一个静态库，磁盘上存了 10 份拷贝

**产出一个静态库的完整命令**：

```bash
# 第一步：编译源码为目标文件
gcc -c hello.c -o hello.o
gcc -c world.c -o world.o

# 第二步：用 ar 打包
ar rcs libhello.a hello.o world.o

# 第三步：验证（查看 .a 中有哪些 .o）
ar t libhello.a
# 输出：hello.o  world.o
```

`ar rcs` 的三个参数：
- `r`：插入文件到归档中（替换已存在的同名文件）
- `c`：如果归档文件不存在则创建
- `s`：写入目标文件索引（让链接器能快速定位符号在哪个 .o 中）

## 1.3 动态库（.so / .dll / .dylib）

**本质**：把多个 .o 文件通过链接器（ld）**链接成一个独立的可加载镜像**。动态库本身是一个完整的 ELF（Linux）或 PE（Windows）文件。

```
libhello.so = hello.o + world.o + utils.o  （ld -shared 链接）
```

**链接行为**：链接器在链接可执行文件时，只做两件事：
1. 在可执行文件中记录"我需要 libhello.so"
2. 把用到的外部符号标记为"运行时去 .so 里找"

**运行时行为**：操作系统加载可执行文件时，动态链接器（ld-linux.so）读取可执行文件中的依赖记录，找到 libhello.so，把它加载进进程的地址空间，然后修正符号地址。

**结果**：
- 可执行文件不包含库代码 → 文件体积小
- 运行时必须有 .so 文件 → 部署需要带 .so
- 10 个程序可以共享同一份 .so（内存中只加载一份代码段）
- .so 可以独立升级，不重新编译依赖它的程序（只要 ABI 不变）

**产出一个动态库的完整命令**：

```bash
# 第一步：编译为位置无关代码（PIC）
gcc -c -fPIC hello.c -o hello.o
gcc -c -fPIC world.c -o world.o

# 第二步：链接为动态库
gcc -shared -o libhello.so hello.o world.o

# 第三步：验证
file libhello.so
# 输出：libhello.so: ELF 64-bit LSB shared object, x86-64...
```

## 1.4 关键对比

| | 静态库 .a | 动态库 .so |
|---|---|---|
| 创建工具 | ar（打包） | ld（链接） |
| 文件格式 | ar 归档格式 | ELF 可加载镜像 |
| 是否可独立加载 | 否 | 是 |
| 链接时发生了什么 | 代码拷贝进可执行文件 | 记录依赖 + 未解析符号 |
| 运行时需要吗 | 不需要 | 必须存在 |
| 多程序共享 | 否（各有一份） | 是（代码段共享） |
| 升级库 | 需重新链接所有程序 | 替换 .so 即可（ABI 兼容前提下） |

## 1.5 本章小结

静态库的本质是 .o 打包，动态库的本质是 .o 链接成一个可加载的镜像文件。理解了这两个概念后，下一章我们动手写一个最小的库，用 gcc 命令手动走完 .c → .o → .a/.so → 链接可执行程序的全过程。

**为什么从手写命令开始而不是直接上 CMake？** CMake 屏蔽了底层命令，直接学 CMake 只能记住语法但不知道它在干什么。手写一遍命令，后面看 CMakeLists.txt 时能用底层命令去理解每一行 CMake 指令的实际效果。

# 第二章：手写命令从零构建

## 2.1 编写一个最小库：libhello

创建以下三个文件：

**hello.h** —— 公开头文件（给库的使用者看的）

```c
#ifndef HELLO_H
#define HELLO_H

// 打印问候语
void hello_say(const char *name);

// 返回版本号
int hello_version(void);

#endif
```

**hello.c** —— 库的实现

```c
#include <stdio.h>
#include "hello.h"

void hello_say(const char *name) {
    printf("Hello, %s!\n", name);
}

int hello_version(void) {
    return 1;
}
```

**main.c** —— 库的使用者

```c
#include "hello.h"

int main() {
    hello_say("world");
    return 0;
}
```

## 2.2 构建静态库并链接

```bash
# 第一步：编译库的源文件为 .o
gcc -c hello.c -o hello.o
# 产物：hello.o

# 第二步：打包为静态库
ar rcs libhello.a hello.o
# 产物：libhello.a

# 第三步：编译 main.c 为 .o（此时不需要库，只需要头文件）
gcc -c main.c -o main.o
# 产物：main.o

# 第四步：链接为可执行文件
gcc main.o -L. -lhello -o main_static
# -L.   ：在当前目录搜索库文件
# -lhello：链接 libhello.a（去掉 lib 前缀和 .a 后缀）
# 产物：main_static

# 运行
./main_static
# 输出：Hello, world!
```

用 `file` 和 `ls -l` 看产物：

```bash
file main_static
# main_static: ELF 64-bit LSB executable, dynamically linked...
# 注意：这里写的是 "dynamically linked"，因为链接了 libc.so。
# 但 libhello.a 的代码是静态链接进 main_static 的。

ls -l main_static
# 大小约 16KB（包含 libhello 和 printf 的 PLT）
```

验证静态库链接后不再需要 .a 文件——删掉 libhello.a，main_static 依然能运行：

```bash
rm libhello.a
./main_static         # 正常运行
```

## 2.3 构建动态库并链接

```bash
# 第一步：编译为位置无关代码
gcc -c -fPIC hello.c -o hello_pic.o
# -fPIC：Position Independent Code，生成位置无关代码

# 第二步：链接为动态库
gcc -shared -o libhello.so hello_pic.o
# -shared：告诉链接器产出动态库而非可执行文件
# 产物：libhello.so

# 第三步：编译 main.c
gcc -c main.c -o main.o

# 第四步：链接可执行文件（这次链接 libhello.so）
gcc main.o -L. -lhello -o main_dynamic
# 链接器发现 libhello.so 和 libhello.a 同时存在时，优先使用 .so
```

验证动态库的必要性——删掉 libhello.so 后无法运行：

```bash
rm libhello.so
./main_dynamic
# 报错：error while loading shared libraries: libhello.so: cannot open shared object file
```

把 libhello.so 放回来并设置搜索路径：

```bash
# 方式一：用 LD_LIBRARY_PATH
LD_LIBRARY_PATH=. ./main_dynamic    # 正常运行

# 方式二：编译时写入 RPATH
gcc main.o -L. -lhello -Wl,-rpath,/home/user/mylib -o main_dynamic
# -Wl,-rpath,/path 告诉链接器把路径写入可执行文件的 RPATH 字段
```

## 2.4 -fPIC 为什么重要

没有 -fPIC 时，.o 文件中的函数地址是**绝对地址**——编译时就假设"我将来被加载到地址 A"。多个 .o 链接成一个可执行文件时，链接器可以调整这些绝对地址（重定位）。

但动态库被加载到进程地址空间的哪个位置，编译时完全不知道。如果多个进程加载同一个 .so，每个进程中 .so 的加载地址可能不同。如果用绝对地址，就必须为每个进程单独做地址修正，这意味着 .so 的代码段不能共享。

加上 -fPIC 后，生成的代码通过**全局偏移表**（GOT）和**过程链接表**（PLT）间接访问所有全局符号和函数。无论 .so 被加载到哪里，只需修改 GOT 中的指针值，代码段本身的指令不需要改变。这样所有进程可以共享同一份代码段的物理内存页。

```bash
# 验证：不带 -fPIC 编译的 .o，链接动态库时会报错（某些架构）
gcc -c hello.c -o hello.o        # 无 -fPIC
gcc -shared -o libhello.so hello.o
# 在 x86-64 上可能不报错，但在 ARM 上会报重定位错误
# 标准做法：编译动态库的 .o 时始终加 -fPIC
```

## 2.5 链接器如何选择静态库 vs 动态库

当 `gcc -L. -lhello` 时，链接器在当前目录搜索：

1. `libhello.so` → 找到了，动态链接
2. `libhello.a` → .so 没找到时，静态链接

如果两者都存在，默认优先 .so。强制使用 .a：

```bash
gcc main.o -L. -l:libhello.a -o main_static    # -l: 指定完整文件名
# 或
gcc main.o -L. -Wl,-Bstatic -lhello -o main_static  # 强制静态链接
```

## 2.6 本章小结

这一章用三个源文件走完了 libhello 从源码到 .a / .so 的完整构建链路。核心记忆点：

- 静态库 = `gcc -c` + `ar rcs`
- 动态库 = `gcc -c -fPIC` + `gcc -shared`
- 使用者链接 = `gcc -L<路径> -l<库名去掉lib前缀>`

**但手写命令只适合教学**——实际项目有几十上百个源文件、多个依赖库、需要同时产出静态库和动态库、需要安装到系统、需要适配不同平台。这些复杂度手工管理会崩溃。所以下一章先定义好库工程的目录结构，然后在第四章用 CMake 管理构建。

# 第三章：库的工程目录结构

## 3.1 标准布局

一个规范的私有库工程遵循以下目录结构：

```
mylib/
├── include/              # 公开头文件（使用者需要的）
│   └── mylib/
│       ├── mylib.h       # 主头文件（一个 include 引入全部）
│       ├── api_a.h       # 模块A的接口
│       └── api_b.h       # 模块B的接口
├── src/                  # 源码（使用者不需要看的）
│   ├── api_a.c
│   ├── api_b.c
│   ├── internal.h        # 内部头文件（不安装到系统）
│   └── internal_impl.c
├── CMakeLists.txt        # 构建脚本
├── README.md             # 使用说明
└── VERSION               # 语义化版本号
```

这里有两个关键设计，逐一说明。

## 3.2 为什么头文件放在 include/mylib/ 而不是 include/

看两个 #include 写法：

```c
// 方案A：头文件直接在 include/ 下
#include "mylib.h"
#include "api_a.h"

// 方案B：头文件在 include/mylib/ 下
#include <mylib/mylib.h>
#include <mylib/api_a.h>
```

方案 B 更好的原因：

1. **命名空间隔离**：如果两个库都有 `config.h`，方案A产生冲突，方案B不会（`mylib/config.h` vs `yourlib/config.h`）
2. **安装友好**：`cmake --install` 把 `include/mylib/` 整目录拷贝到 `/usr/include/mylib/`，不会污染全局 include
3. **清晰表明来源**：读代码时一眼知道这个头文件来自哪个库

mosquitto 就是这个做法——`#include <mosquitto.h>`，安装后 `mosquitto.h` 在 `/usr/include/mosquitto.h`（单个文件的情况也可以不加子目录，取决于复杂度）。

## 3.3 公开头文件 vs 内部头文件

- **公开头文件**（include/ 下）：声明库对外提供的 API——函数签名、结构体、常量、宏。只包含让使用者"调用"所需的最小信息
- **内部头文件**（src/ 下）：声明库内部使用的类型、函数、宏——编译时需要但不暴露给使用者

**区分的好处**：
- 库的使用者只需要看 include/ 下的文件就能使用库，不必翻源码
- 改动内部实现不重新编译使用者代码（接口没变时）
- 安装时只装 include/ 下的头文件

## 3.4 版本号与 soname 机制

动态库有三级命名：

| 层 | 名称 | 示例 | 作用 |
|---|---|---|---|
| real name | 真实文件名 | `libhello.so.1.0.0` | 实际存储的文件，MAJOR.MINOR.PATCH |
| soname | 链接器记录的名称 | `libhello.so.1` | 包含在 .so 的 ELF 头中，运行时按这个找 |
| linker name | 编译时用的名称 | `libhello.so` | 编译时 `-lhello` 找到的符号链接 |

它们之间的关系靠**符号链接**：

```
libhello.so        → libhello.so.1       （linker name → soname）
libhello.so.1      → libhello.so.1.0.0   （soname → real name）
libhello.so.1.0.0                          （真实文件）
```

**MAJOR 版本变化（1→2）**：API 不兼容，需要重新编译依赖库的程序。soname 变化（libhello.so.1 → libhello.so.2），旧程序无法加载新版 .so。

**MINOR 版本变化（0→1）**：新增 API，旧 API 不变。soname 不变（仍是 libhello.so.1），旧程序可以加载新版 .so。

**PATCH 版本变化（0→1）**：只修 bug，不改变 API。直接替换 real name 文件。

## 3.5 本章小结

目录结构、头文件分层、版本号命名——这三样定下来之后，就有了一个库工程的骨架。下一章用 CMake 把这个骨架变成可构建、可安装的真实工程。

**为什么先定结构再写 CMake？** CMakeLists.txt 中的路径设置（include 目录在哪里、源文件在哪里、安装目标路径是什么）直接映射到这个目录结构。结构明确了，CMake 脚本就是对这个结构的机器可读描述。

# 第四章：CMake 构建脚本

## 4.1 最小 CMakeLists.txt

对应第三章的目录结构，一个同时产出静态库和动态库的 CMakeLists.txt：

```cmake
cmake_minimum_required(VERSION 3.10)
project(mylib VERSION 1.0.0 LANGUAGES C)

# ============================================================
# 第一步：收集源文件
# ============================================================
set(SOURCES
    src/api_a.c
    src/api_b.c
    src/internal_impl.c
)

# ============================================================
# 第二步：构建静态库
# ============================================================
add_library(mylib_static STATIC ${SOURCES})
target_include_directories(mylib_static
    PUBLIC
        $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
        $<INSTALL_INTERFACE:include>
    PRIVATE
        ${CMAKE_CURRENT_SOURCE_DIR}/src
)

# ============================================================
# 第三步：构建动态库
# ============================================================
add_library(mylib_shared SHARED ${SOURCES})
target_include_directories(mylib_shared
    PUBLIC
        $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
        $<INSTALL_INTERFACE:include>
    PRIVATE
        ${CMAKE_CURRENT_SOURCE_DIR}/src
)

# 设置 soname（动态库专属）
set_target_properties(mylib_shared PROPERTIES
    VERSION ${PROJECT_VERSION}      # 1.0.0 → real name
    SOVERSION 1                     # .1     → soname
)

# ============================================================
# 第四步：安装规则
# ============================================================
install(TARGETS mylib_static mylib_shared
    LIBRARY DESTINATION lib       # .so 装到 lib/
    ARCHIVE DESTINATION lib       # .a 装到 lib/
)
install(DIRECTORY include/mylib
    DESTINATION include           # 公开头文件装到 include/mylib/
)
```

## 4.2 逐段解释

### add_library

```cmake
add_library(mylib_static STATIC ${SOURCES})
add_library(mylib_shared SHARED ${SOURCES})
```

这两个命令各创建一个构建目标。`STATIC` 对应 `ar rcs`，`SHARED` 对应 `gcc -shared -fPIC`。CMake 自动在 SHARED 模式下给编译器加 `-fPIC`。

### target_include_directories 的 PUBLIC vs PRIVATE

```cmake
target_include_directories(mylib_shared
    PUBLIC
        $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
        $<INSTALL_INTERFACE:include>
    PRIVATE
        ${CMAKE_CURRENT_SOURCE_DIR}/src
)
```

这里用到了 CMake 的**传递性依赖**概念：

- **PUBLIC**：include/mylib 是公开头文件路径。链接 mylib 的目标自动获得这个 include 路径。对应 `#include <mylib/mylib.h>` 这种公开 API
- **PRIVATE**：src/ 是内部源码路径。只有 mylib 自己编译时需要，链接 mylib 的目标不会获得这个路径。对应 `#include "internal.h"` 这种内部头文件

`$<BUILD_INTERFACE:...>` 和 `$<INSTALL_INTERFACE:...>` 是生成器表达式——编译时的 include 路径和安装后的 include 路径不同，编译时指向源码树中的 `include/`，安装后指向 `/usr/include/`。

### 版本属性

```cmake
set_target_properties(mylib_shared PROPERTIES
    VERSION ${PROJECT_VERSION}   # CMake 用 project(VERSION 1.0.0) 设置
    SOVERSION 1
)
```

产生如下文件：
```
libmylib.so → libmylib.so.1 → libmylib.so.1.0.0
```
VERSION 是 real name 中的版本号，SOVERSION 是 soname 中的主版本号。

### install 规则

```cmake
install(TARGETS mylib_static mylib_shared
    LIBRARY DESTINATION lib      # SHARED 库 → lib/
    ARCHIVE DESTINATION lib      # STATIC 库 → lib/
)
install(DIRECTORY include/mylib
    DESTINATION include          # 头文件 → include/mylib/
)
```

执行 `cmake --install . --prefix /usr/local` 后的结果：

```
/usr/local/
├── include/mylib/          # 公开头文件
│   ├── mylib.h
│   ├── api_a.h
│   └── api_b.h
└── lib/
    ├── libmylib.a           # 静态库
    ├── libmylib.so.1.0.0    # 动态库 (real name)
    ├── libmylib.so.1 → libmylib.so.1.0.0   # soname 符号链接
    └── libmylib.so → libmylib.so.1          # linker name 符号链接
```

## 4.3 完整的构建流程

```bash
# 配置（在 build 目录中，保持源码树干净）
mkdir build && cd build
cmake .. -DCMAKE_INSTALL_PREFIX=/usr/local

# 编译
cmake --build .

# 安装（可能需要 sudo）
cmake --install .
```

## 4.4 其他项目如何链接我们的库

其他 CMake 项目使用我们的库：

```cmake
# 使用者的 CMakeLists.txt
find_package(mylib REQUIRED)     # 需要写一个 mylibConfig.cmake（见 4.6）
target_link_libraries(myapp PRIVATE mylib::mylib_shared)
```

或者传统方式：

```bash
gcc main.c -I/usr/local/include -L/usr/local/lib -lmylib -o main
```

## 4.5 同一个 CMake 工程中也产出可执行测试程序

在同一个 CMakeLists.txt 中追加：

```cmake
# 测试程序（不安装）
add_executable(test_mylib test/main.c)
target_link_libraries(test_mylib PRIVATE mylib_shared)
```

测试程序链接动态库，验证 API 正确性。`PRIVATE` 表示 test_mylib 不对外暴露 mylib 的头文件路径。

## 4.6 让其他 CMake 项目用 find_package 找到我们的库

需要提供一个 `mylibConfig.cmake` 文件：

```cmake
# mylibConfig.cmake（安装到 lib/cmake/mylib/）
include("${CMAKE_CURRENT_LIST_DIR}/mylibTargets.cmake")
```

以及对应的安装规则：

```cmake
# 在 CMakeLists.txt 中追加
include(CMakePackageConfigHelpers)

install(EXPORT mylibTargets
    FILE mylibTargets.cmake
    NAMESPACE mylib::
    DESTINATION lib/cmake/mylib
)

install(FILES ${CMAKE_CURRENT_BINARY_DIR}/mylibConfig.cmake
    DESTINATION lib/cmake/mylib
)
```

这样其他项目 `find_package(mylib)` 时，CMake 自动找到我们的库并导入目标 `mylib::mylib_shared` 和 `mylib::mylib_static`。

## 4.7 本章小结

一个完整的 CMakeLists.txt 覆盖了：定义目标 → 设置 include 路径 → 区分公开/私有一→ 设置版本号 → 安装规则 → 导出 CMake 包。

**但有一个问题还没解决**：这个库编译出来后，所有非 static 函数都被导出到 .so 的符号表中。使用者调 API 没问题，但像 `internal_impl.c` 中的内部函数也暴露了——这带来三个问题：
1. 使用者可能误用内部函数（以为它们是公开 API）
2. 增加符号表体积，拖慢动态加载
3. 内部函数改动后破坏符号兼容性

下一章解决这个问题——控制符号可见性，只导出真正想公开的符号。

# 第五章：符号可见性控制

## 5.1 问题演示

先用 nm 命令查看上一章产出的 libmylib.so 导出了哪些符号：

```bash
nm -D libmylib.so | grep " T "
# T 表示代码段的导出符号
# 输出：
# 0000000000001140 T api_a_public_func
# 0000000000001180 T api_b_public_func
# 00000000000011c0 T internal_helper     ← 内部函数被导出
# 0000000000001200 T internal_parse      ← 内部函数被导出
```

所有非 static 函数都在符号表中。我们只希望 `api_a_public_func` 和 `api_b_public_func` 对外可见。

## 5.2 GCC/Clang 的 visibility 机制

GCC 4.0+ 提供 `__attribute__((visibility(...)))` 控制单个符号的可见性：

```c
// 导出（默认）
__attribute__((visibility("default")))
void api_a_public_func(void);

// 隐藏
__attribute__((visibility("hidden")))
void internal_helper(void);
```

但每个函数写一长串不现实。标准做法是**编译时默认隐藏，手动标记要导出的**。

**第一步**：加编译选项 `-fvisibility=hidden`，让所有符号默认不导出。

**第二步**：在公开头文件中定义导出宏：

```c
// include/mylib/mylib.h
#ifndef MYLIB_H
#define MYLIB_H

// 导出宏：根据编译环境自动切换
#if defined _WIN32 || defined __CYGWIN__
    // Windows: 用 __declspec
    #ifdef MYLIB_BUILDING
        #define MYLIB_EXPORT __declspec(dllexport)
    #else
        #define MYLIB_EXPORT __declspec(dllimport)
    #endif
#else
    // Linux: 用 visibility
    #if __GNUC__ >= 4
        #define MYLIB_EXPORT __attribute__((visibility("default")))
    #else
        #define MYLIB_EXPORT
    #endif
#endif

// 公开 API 前加导出宏
MYLIB_EXPORT void hello_say(const char *name);
MYLIB_EXPORT int hello_version(void);

#endif
```

**第三步**：源码中定义 `MYLIB_BUILDING` 宏（编译库自身时）：

```cmake
# CMakeLists.txt 中
target_compile_definitions(mylib_shared PRIVATE MYLIB_BUILDING)
target_compile_options(mylib_shared PRIVATE -fvisibility=hidden)
```

`MYLIB_BUILDING` 只在编译库时定义，使用者不定义。所以编译库时 `MYLIB_EXPORT` 展开为 `__attribute__((visibility("default")))`（dllexport），使用者编译时展开为 `__declspec(dllimport)`（或 Linux 下同 `visibility("default")`）。

## 5.3 验证效果

```bash
# 编译后检查
nm -D libmylib.so | grep " T "
# 现在只有标记了 MYLIB_EXPORT 的函数出现：
# 0000000000001140 T api_a_public_func
# 0000000000001180 T api_b_public_func
# internal_helper 和 internal_parse 不见了
```

## 5.4 用 CMake 自动生成导出宏

CMake 的 `GenerateExportHeader` 模块可以自动生成导出头文件，比手写更可靠：

```cmake
include(GenerateExportHeader)

add_library(mylib_shared SHARED ${SOURCES})
generate_export_header(mylib_shared
    BASE_NAME mylib
    EXPORT_FILE_NAME ${CMAKE_CURRENT_BINARY_DIR}/include/mylib/mylib_export.h
)
target_include_directories(mylib_shared
    PUBLIC ${CMAKE_CURRENT_BINARY_DIR}/include
)
```

CMake 自动生成 `mylib_export.h`：

```c
// 自动生成的 mylib_export.h（内容简化）
#ifndef MYLIB_EXPORT_H
#define MYLIB_EXPORT_H

#ifdef MYLIB_STATIC_DEFINE
#  define MYLIB_EXPORT
#else
#  ifdef mylib_shared_EXPORTS    // CMake 自动定义此宏
#    define MYLIB_EXPORT __attribute__((visibility("default")))
#  else
#    define MYLIB_EXPORT          // 使用者编译时不加属性
#  endif
#endif

#endif
```

使用者在代码中直接用：

```c
#include "mylib/mylib_export.h"
#include "mylib/mylib.h"
```

## 5.5 本章小结

符号可见性控制的三个步骤：
1. 编译选项默认隐藏（`-fvisibility=hidden`）
2. 导出宏标记公开 API（`MYLIB_EXPORT`）
3. CMake 编译定义区分编译库 vs 使用库（`MYLIB_BUILDING`）

这一章让库的二进制接口变得干净——从"全部暴露"变成"只暴露想暴露的"。接下来处理最后一个实际问题：库编译安装好了，怎么让其他程序在运行时找到它。

# 第六章：安装与运行时查找

## 6.1 动态库的运行时搜索顺序

动态链接器（ld-linux.so）加载可执行文件时，按以下顺序搜索 .so：

```
1. RPATH          —— 编译进 ELF 的路径（传统方式，正在被 RUNPATH 取代）
2. RUNPATH        —— 编译进 ELF 的路径（现代方式，比 RPATH 灵活）
3. LD_LIBRARY_PATH —— 环境变量（优先级高于系统路径，适合临时测试）
4. /etc/ld.so.cache —— ldconfig 缓存（系统标准路径的索引）
5. /lib 和 /usr/lib —— 默认系统路径
```

用 `readelf` 查看可执行文件的 RPATH/RUNPATH：

```bash
readelf -d ./main_dynamic | grep -E "RPATH|RUNPATH"
# 0x000000000000001d (RUNPATH)  Library runpath: [/usr/local/lib]
```

用 `ldd` 查看可执行文件能不能找到所有 .so：

```bash
ldd ./main_dynamic
# libhello.so => /usr/local/lib/libhello.so (0x00007f...)
# 如果某个库显示 "not found"，运行时就会报错
```

## 6.2 三种部署策略

### 策略一：安装到系统路径

```bash
sudo cmake --install . --prefix /usr/local
sudo ldconfig                  # 更新动态链接器缓存
```

优点：所有程序都能找到，不需要额外配置。适用场景：系统级库（像 libmosquitto.so 就装在 /usr/local/lib）。

### 策略二：RPATH 写入可执行文件

```cmake
# 在 CMakeLists.txt 中设置 RPATH
set_target_properties(myapp PROPERTIES
    INSTALL_RPATH "${CMAKE_INSTALL_PREFIX}/lib"
)
```

编译时将 `/usr/local/lib` 写入可执行文件的 RPATH。运行时动态链接器优先去这个路径找 .so。适用场景：库和可执行文件一起安装到固定前缀。

### 策略三：随应用打包（$ORIGIN）

在 RPATH 中使用 `$ORIGIN` 变量——动态链接器将其展开为可执行文件所在目录：

```cmake
set_target_properties(myapp PROPERTIES
    INSTALL_RPATH "$ORIGIN/../lib"
)
```

假设安装后：
```
/opt/myapp/
├── bin/myapp              # 可执行文件
└── lib/libmylib.so        # 库文件
```

$ORIGIN 展开为 `/opt/myapp/bin`，`$ORIGIN/../lib` 即为 `/opt/myapp/lib`。这样不管安装到哪里，库都能找到，适合分发给用户的二进制包。

## 6.3 验证与排查

常用排查工具：

```bash
# 查看依赖哪些 .so，以及是否全部找到
ldd /path/to/executable

# 查看 .so 中导出哪些符号
nm -D /path/to/lib.so

# 查看可执行文件的 RPATH/RUNPATH
readelf -d /path/to/executable | grep -E "RPATH|RUNPATH"

# 查看 .so 的 soname
readelf -d /path/to/lib.so | grep SONAME

# 列出 .a 中包含哪些 .o
ar t /path/to/lib.a
```

## 6.4 本章小结

配置好 CMake 安装规则 → 选好运行时查找策略 → 用 ldd/readelf 验证 → 私有库从构建到部署的完整链路打通。

# 第七章：跨平台适配要点

## 7.1 .so (Linux) 与 .dll (Windows) 的关键差异

| | Linux .so | Windows .dll |
|---|---|---|
| 符号导出策略 | 默认全部导出，通过 visibility 限制 | 默认全部隐藏，必须显式 `__declspec(dllexport)` |
| 导入声明 | 不需要 | 使用者需 `__declspec(dllimport)` |
| 运行时搜索 | RPATH/RUNPATH/LD_LIBRARY_PATH/ldconfig | PATH 环境变量 + 可执行文件所在目录 |
| 产物命名 | libxxx.so | xxx.dll（无 lib 前缀） |
| 导入库 | 不需要（.so 直接用于链接） | 需要 .lib 导入库（不是静态库，是 DLL 的索引） |
| 静态库命名 | libxxx.a | xxx.lib（易与导入库混淆） |

## 7.2 CMake 自动处理平台差异

CMake 的关键价值在于——同一份 CMakeLists.txt，Linux 上产出 libhello.so，Windows 上产出 hello.dll：

```cmake
add_library(mylib_shared SHARED ${SOURCES})
# Linux: CMake 生成 gcc -shared -fPIC -o libmylib.so
# Windows: CMake 生成 cl /LD /Fe:mylib.dll 并产生 mylib.lib（导入库）
```

`GenerateExportHeader` 自动处理 `__declspec(dllexport/dllimport)` vs `visibility("default")`。

## 7.3 Windows 特有的注意事项

- **符号导出**：没有 `-fvisibility=hidden` 等价物，默认全部隐藏。`GenerateExportHeader` 自动生成正确的 `__declspec` 宏
- **.lib 导入库**：链接 .dll 时需要 .lib 导入库，它不是静态库——它只包含符号索引和跳转桩，不包含函数体。CMake 的 SHARED 库在 Windows 上自动生成这个导入库
- **运行时搜索**：把 .dll 放在与 .exe 同目录，或 PATH 中的路径。没有 RPATH 机制
- **DLL 中定义全局变量**：需要特殊处理（`__declspec(dllimport)` 在变量上），否则链接器产生错误的重定位

## 7.4 本章小结

跨平台适配的核心策略：**把平台差异交给 CMake 和 GenerateExportHeader 处理**，源码中只使用统一的导出宏。极端情况下需要 `#ifdef _WIN32` 时，把这些条件编译限定在平台适配层，不要散落在业务逻辑中。

# 第八章：参考案例——mosquitto 的构建脚本

## 8.1 mosquitto 的目录结构

从 CameraTest 工程的 `3rdparty/mqtt/` 中找到 mosquitto 源码（或 GitHub 上的官方仓库），它的目录布局：

```
mosquitto/
├── include/
│   └── mosquitto.h          # 唯一的公开头文件（800+ 行）
├── lib/
│   ├── mosquitto.c          # MQTT 客户端 API 实现
│   ├── mosquitto_internal.h # 内部头文件（数据包、连接管理）
│   ├── net_mosq.c           # 网络层
│   ├── send_mosq.c          # 发送
│   ├── util_mosq.c          # 工具函数
│   └── ...                  # 共约 20 个 .c 文件
├── src/                     # mosquitto broker 源码（不是库）
├── CMakeLists.txt           # 顶层构建脚本
└── lib/CMakeLists.txt       # 库的构建脚本（核心）
```

注意 mosquitto 的库和 broker 在同一个仓库中，但库有独立的 CMakeLists.txt。

## 8.2 mosquitto 的 lib/CMakeLists.txt（核心部分）

```cmake
# 省略了 OpenSSL 检测、编译选项等
set(MOSQ_SRCS
    mosquitto.c
    actions.c
    callbacks.c
    connect.c
    handle_auth.c
    handle_connack.c
    # ... 约 20 个源文件
)

# 动态库
add_library(mosquitto SHARED ${MOSQ_SRCS})
set_target_properties(mosquitto PROPERTIES
    VERSION ${VERSION}          # 2.0.15 之类
    SOVERSION 1
)
target_include_directories(mosquitto
    PUBLIC
        $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/../include>
        $<INSTALL_INTERFACE:include>
    PRIVATE
        ${CMAKE_CURRENT_SOURCE_DIR}
)
target_link_libraries(mosquitto PUBLIC ${OPENSSL_LIBRARIES})

# 静态库
add_library(mosquitto_static STATIC ${MOSQ_SRCS})
target_include_directories(mosquitto_static
    PUBLIC
        $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/../include>
        $<INSTALL_INTERFACE:include>
    PRIVATE
        ${CMAKE_CURRENT_SOURCE_DIR}
)
target_link_libraries(mosquitto_static PUBLIC ${OPENSSL_STATIC_LIBRARIES})

# 安装
install(TARGETS mosquitto mosquitto_static
    LIBRARY DESTINATION ${CMAKE_INSTALL_LIBDIR}
    ARCHIVE DESTINATION ${CMAKE_INSTALL_LIBDIR}
)
install(FILES ../include/mosquitto.h
    DESTINATION ${CMAKE_INSTALL_INCLUDEDIR}
)
```

## 8.3 与我们的 CMakeLists.txt 对照

对比第四章我们自己写的 CMakeLists.txt 和 mosquitto 的真实脚本，结构完全一致：

| 步骤 | 我们的私有库 | mosquitto |
|------|------------|-----------|
| 源文件收集 | `set(SOURCES src/... )` | `set(MOSQ_SRCS mosquitto.c ...)` |
| 动态库 | `add_library(mylib_shared SHARED ...)` | `add_library(mosquitto SHARED ...)` |
| 版本号 | `VERSION 1.0.0` `SOVERSION 1` | `VERSION ${VERSION}` `SOVERSION 1` |
| 公开头文件 | `PUBLIC .../include` | `PUBLIC .../include` |
| 内部头文件 | `PRIVATE .../src` | `PRIVATE .` (当前lib目录) |
| 外部依赖 | 无 | `target_link_libraries(mosquitto PUBLIC ${OPENSSL_LIBRARIES})` |
| 安装 | `install(TARGETS ...)` | 相同 |

mosquitto 比我们多的一步是**外部依赖管理**——`target_link_libraries(mosquitto PUBLIC ${OPENSSL_LIBRARIES})`。这里的 `PUBLIC` 表示"链接 mosquitto 的程序也自动链接 OpenSSL"，因为 mosquitto.h 中的公开 API 参数用到了 OpenSSL 的 `SSL_CTX` 类型。

## 8.4 一个外部依赖的处理示例

如果你开发的私有库依赖了 mosquitto，需要这样声明：

```cmake
# 假设我们的 mylib 使用了 mosquitto 的 API
find_package(mosquitto REQUIRED)

add_library(mylib_shared SHARED ${SOURCES})

# 依赖分为两种：
# PRIVATE —— mosquitto 只在 mylib 内部使用，不暴露给使用者
target_link_libraries(mylib_shared PRIVATE mosquitto::mosquitto)

# PUBLIC —— mylib 公开 API 中包含 mosquitto 类型（如 mosquitto.h 中的结构体）
target_link_libraries(mylib_shared PUBLIC mosquitto::mosquitto)
```

PRIVATE 的依赖不会传递给 `target_link_libraries(myapp PRIVATE mylib_shared)` 的 myapp。PUBLIC 会传递。

# 全篇总结

从零构建私有库的完整链路：

```
源码文件(.c/.h)
    │
    ├── 头文件分层：include/ 公开  vs  src/ 内部
    │
    ├── gcc -c → .o（加了 -fPIC 就是位置无关代码）
    │       │
    │       ├── ar rcs → .a（静态库：.o 的归档包）
    │       │
    │       └── gcc -shared → .so（动态库：完整的可加载镜像）
    │
    ├── CMakeLists.txt 管理以上过程
    │       ├── add_library(STATIC ...)
    │       ├── add_library(SHARED ...)
    │       ├── target_include_directories (PUBLIC/PRIVATE)
    │       └── install(...)
    │
    ├── 符号可见性控制
    │       ├── -fvisibility=hidden（默认隐藏）
    │       └── MYLIB_EXPORT 宏（标记公开API）
    │
    └── 部署
            ├── cmake --install → /usr/local/
            ├── RPATH / $ORIGIN（运行时搜索路径）
            └── ldconfig（系统缓存更新）
```

各章之间的依赖关系：
- 第一章定义概念（库是什么）
- 第二章手写命令验证概念（底层怎么做）
- 第三章定义工程结构（怎么组织文件）
- 第四章用 CMake 描述结构（怎么自动化）

- 第六章处理部署（怎么让别人用）
- 第七章适配平台差异（怎么跨平台）

# 具体实现过程
### 用静态库 (.a)

```bash
# 编译 main.c（只需要头文件，不需要库）
gcc -c main.c -I./include -o main.o

# 链接：把 main.o 和 libhello.a 合成可执行文件
gcc main.o -L./lib -lhello -o main

# 运行
./main
# 输出：Hello, world!
```

链接这一步，链接器做的事情：

1. 读到 `main.o` 中有一个未定义的符号 `hello_say`
2. 去 `-L./lib` 指定的目录找 `libhello.a`
3. 在 `libhello.a` 里找到 `hello.o` 提供了 `hello_say`
4. 把 `hello.o` 从 .a 中**提取出来，拷贝进 `main` 可执行文件**
5. 修正 `main.o` 中调用 `hello_say` 的地址

完成之后，`main` 这个可执行文件**自带了 `hello_say` 的完整机器码**，不再需要 `libhello.a`。你可以删掉 .a，main 照样运行。

### 用动态库 (.so)

```bash
# 编译 main.c（和静态库一样）
gcc -c main.c -I./include -o main.o

# 链接：链接器不拷贝代码，只记录"我需要 libhello.so"
gcc main.o -L./lib -lhello -o main

# 运行——此时必须能找到 libhello.so
LD_LIBRARY_PATH=./lib ./main
# 输出：Hello, world!
```

链接这一步，链接器的行为**完全不同**：

1. 发现 `main.o` 需要 `hello_say`
2. 在 `libhello.so` 中找到 `hello_say` 这个符号
3. **不拷贝代码**，只在 `main` 可执行文件里写一条记录："运行时去 `libhello.so` 找 `hello_say`"
4. `main` 文件体积比静态链接小得多

运行时：

1. 操作系统加载 `main`
2. 动态链接器读到 `main` 的依赖记录，找到 `libhello.so`
3. 把 `libhello.so` 加载进内存
4. 修正 `main` 中 `hello_say` 的调用地址，指向内存中 `libhello.so` 的对应位置

删掉 `libhello.so`，main 启动时直接报错退出。
