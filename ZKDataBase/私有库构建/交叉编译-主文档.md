# 交叉编译-主文档

## 本文主线

在 x86-64 PC 上编译出 ARM 板子能运行的库，全程以之前做库的流程为参照，标注交叉编译改变的环节，用 CMake toolchain file 管理工具链切换。

---

# 第一章：交叉编译的本质

**交叉编译 = 在一种架构的机器上，编译出另一种架构上能运行的二进制文件。**

```
本地编译：  x86-64 的 gcc + x86-64 的 ld → 产出 x86-64 的二进制
交叉编译：  x86-64 上运行的 arm-gcc + arm-ld → 产出 ARM 的二进制
```

交叉编译器本身跑在 PC 上（x86-64），产出的机器码是 ARM 指令集。运行环境和目标环境是两个不同的 CPU 架构。

---

# 第二章：做库流程中哪几步变了

回顾之前做库的七个步骤，标注受影响的环节：

```
1. 整理目录结构        → 不变（源码和 .h 跟架构无关）
2. 检查调整头文件      → 不变（C 语言的 .h 是架构无关的）
3. 写 CMakeLists.txt   → 基本不变（多加一个 toolchain file）
4. 编译 ★              → 工具换了：gcc 换成 arm-linux-gnueabihf-gcc
5. 链接 ★              → 工具换了：ar 换成 arm-linux-gnueabihf-ar
                         ld 换成 arm-linux-gnueabihf-ld
6. 验证 ★              → 变了：产物不能在 PC 上运行，要拷到 ARM 板子
7. 在其他项目中使用    → 不变：头文件用法一样，链接命令逻辑一样
```

**核心变化只在第 4、5、6 步——编译链接的工具换了，验证方式从"本地跑"变成"拷到板子上跑"。** 源码不需要因架构改变。

# 第三章：交叉编译工具链

## 3.1 工具链组成

一套完整的 ARM 交叉编译工具链包含：

| 工具 | 本地编译 | ARM 交叉编译 | 作用 |
|------|---------|-------------|------|
| C 编译器 | `gcc` | `arm-linux-gnueabihf-gcc` | .c → .o |
| C++ 编译器 | `g++` | `arm-linux-gnueabihf-g++` | .cpp → .o |
| 静态库打包 | `ar` | `arm-linux-gnueabihf-ar` | .o → .a |
| 链接器 | `ld` | `arm-linux-gnueabihf-ld` | .o → .so / 可执行文件 |

命名规则：`<目标架构>-<目标系统>-<ABI>-<工具名>`

- `arm`：目标 CPU 架构
- `linux`：目标操作系统
- `gnueabihf`：使用 glibc + 硬件浮点 ABI

## 3.2 安装

```bash
# Ubuntu/Debian
sudo apt install gcc-arm-linux-gnueabihf g++-arm-linux-gnueabihf

# 验证安装
arm-linux-gnueabihf-gcc --version
```

# 第四章：手写命令方式

和本地编译的唯一区别——把命令名加上 `arm-linux-gnueabihf-` 前缀。

### 静态库

```bash
# 本地编译
gcc -c hello.c -o hello.o
ar rcs libhello.a hello.o

# 交叉编译（只改了工具名）
arm-linux-gnueabihf-gcc -c hello.c -o hello.o
arm-linux-gnueabihf-ar rcs libhello.a hello.o
```

### 动态库

```bash
# 本地编译
gcc -c -fPIC hello.c -o hello.o
gcc -shared -o libhello.so hello.o

# 交叉编译
arm-linux-gnueabihf-gcc -c -fPIC hello.c -o hello.o
arm-linux-gnueabihf-gcc -shared -o libhello.so hello.o
```

### 验证产物架构

```bash
file hello.o
# hello.o: ELF 32-bit LSB relocatable, ARM, EABI5 version 1 (SYSV)

file libhello.so
# libhello.so: ELF 32-bit LSB shared object, ARM, EABI5 version 1 (SYSV)
```

看到 `ARM` 说明交叉编译成功。如果是 `x86-64` 说明忘换工具链了。

# 第五章：CMake 方式——toolchain file

手写命令只适合演示。实际工程用 CMake + toolchain file 管理。

## 5.1 编写 toolchain file

创建 `arm-toolchain.cmake`：

```cmake
# arm-toolchain.cmake
set(CMAKE_SYSTEM_NAME Linux)                           # 目标系统
set(CMAKE_SYSTEM_PROCESSOR arm)                        # 目标 CPU

# 编译器
set(CMAKE_C_COMPILER   arm-linux-gnueabihf-gcc)       # C 编译器
set(CMAKE_CXX_COMPILER arm-linux-gnueabihf-g++)       # C++ 编译器

# 其他工具（ar、ld 会自动从编译器前缀推导，也可显式指定）
set(CMAKE_AR arm-linux-gnueabihf-ar)

# 目标系统的根文件系统路径（板子的 sysroot 同步到 PC 后）
set(CMAKE_SYSROOT /path/to/arm-rootfs)

# 查找策略：程序用本机的，库和头文件用目标 sysroot 的
set(CMAKE_FIND_ROOT_PATH_MODE_PROGRAM NEVER)   # cmake、make 等工具用本机
set(CMAKE_FIND_ROOT_PATH_MODE_LIBRARY ONLY)    # 库只在 sysroot 找
set(CMAKE_FIND_ROOT_PATH_MODE_INCLUDE ONLY)    # 头文件只在 sysroot 找
set(CMAKE_FIND_ROOT_PATH_MODE_PACKAGE ONLY)    # CMake 包只在 sysroot 找
```

关键点在于 `CMAKE_FIND_ROOT_PATH_MODE` 的四项设置：
- `PROGRAM NEVER`：cmake、make、pkg-config 这类**构建时运行的**程序用本机版本
- `LIBRARY ONLY`：链接时需要的 .so/.a 只从 sysroot 找，不碰本机的 x86-64 库
- `INCLUDE ONLY`：头文件只从 sysroot 找，避免混入本机系统头文件
- `PACKAGE ONLY`：`find_package` 只在 sysroot 搜索

## 5.2 使用 toolchain file 编译

```bash
cd my_uart
mkdir build_arm && cd build_arm

# cmake 配置时指定 toolchain file
cmake .. -DCMAKE_TOOLCHAIN_FILE=../arm-toolchain.cmake

# 编译（CMake 自动调用交叉工具链）
cmake --build .
```

编译完成后检查：

```bash
# 提取 .a 中的 .o 检查架构
ar x libmyuart_static.a
file uart.c.o
# uart.c.o: ELF 32-bit LSB relocatable, ARM, EABI5 version 1 (SYSV)
```

## 5.3 对比：本地编译 vs 交叉编译的 CMake 调用

```bash
# 本地编译（x86-64）
mkdir build && cd build
cmake ..
cmake --build .

# 交叉编译（ARM）
mkdir build_arm && cd build_arm
cmake .. -DCMAKE_TOOLCHAIN_FILE=../arm-toolchain.cmake
cmake --build .
```

唯一的区别是多传了一个 `-DCMAKE_TOOLCHAIN_FILE`。CMakeLists.txt 本身一行不用改。

# 第六章：几个关键差异

## 6.1 运行时验证做不了

本地编译时 `./test_uart` 直接跑。交叉编译产物是 ARM 指令，PC 无法执行：

```bash
./test_uart
# Exec format error（或 cannot execute binary file: Exec format error）
```

替代验证方式：

```bash
file test_uart              # 确认架构是 ARM
readelf -d test_uart        # 确认依赖的 .so 列表
readelf -s libmyuart.so     # 确认导出的符号正确
# 最后拷到 ARM 板子上：scp test_uart root@192.168.1.x:/tmp/ && ssh 运行
```

## 6.2 目标系统的依赖库问题

如果你的库依赖 OpenSSL、mosquitto 等，交叉编译时需要**目标架构版本**的依赖库。这些库在目标板的 sysroot 中：

```
/path/to/arm-rootfs/
├── usr/
│   ├── include/           # ARM 版本的头文件
│   │   └── openssl/
│   └── lib/               # ARM 版本的 .so 和 .a
│       ├── libssl.so
│       └── libcrypto.so
└── lib/
    └── ld-linux-armhf.so.3    # ARM 版本的动态链接器
```

CMake 通过 `CMAKE_SYSROOT` 找到这些文件。`find_package(OpenSSL)` 会去 sysroot 里搜索，不会错拿 PC 上的 x86-64 版本。

## 6.3 倾向静态库

嵌入式 ARM 场景比桌面更倾向静态链接：

| | 本地桌面 | 嵌入式 ARM |
|---|---|---|
| 倾向 | 动态库 .so | 静态库 .a |
| 原因 | 多程序共享、方便升级 | 固件单一、避免板子缺 .so |
| 部署 | .so 随应用或放系统路径 | 全嵌进一个可执行文件 |

CameraTest 工程的 `3rdparty/` 下的 mqtt、jsoncpp 最终大概率是静态链接进 `zksort` 可执行文件的——ARM 板子上没有独立的这些 .so 文件。

## 6.4 ABI 必须一致

工具链名称中的 `gnueabihf` 是 ABI 标识。编译库和编译使用者程序时，ABI 必须一致。如果用 `hf`（硬件浮点）编译了库，使用者的程序也用 `hf`——混用 `hf` 和 `sf`（软件浮点）会导致浮点参数传递方式冲突，链接时符号不匹配或运行时崩溃。

# 第七章：总结

```
本地编译：

    hello.c ──gcc──→ hello.o ──ar──→ libhello.a (x86-64)
                    hello.o ──ld──→ libhello.so (x86-64)
    ./test_uart                     ← 直接跑

交叉编译 ARM：

    hello.c ──arm-gcc──→ hello.o ──arm-ar──→ libhello.a (ARM)
                        hello.o ──arm-ld──→ libhello.so (ARM)
    拷到板子上 ./test_uart                  ← 板子上跑
```

**变的是工具链前缀**（`arm-linux-gnueabihf-`），**不变的是整个构建逻辑**。CMake + toolchain file 把工具链的切换从几十个手动改命令变成传一个文件路径。同一份 CMakeLists.txt，本地和交叉编译各建一个 build 目录，并行存在互不干扰。
