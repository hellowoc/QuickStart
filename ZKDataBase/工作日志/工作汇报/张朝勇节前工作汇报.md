## 工作案例一：工程全貌学习与知识库建设

### 任务背景

入职后，导师谢工提供了 CameraTest 工程源码，要求我自主阅读代码，逐步了解部门的业务方向、代码规范和项目的功能实现，完成从校内"结果导向"思维到商业项目"应用导向"思维的转变。同时，导师也借此评估我的技术水平，以便后续安排针对性的培养计划。

### 学习思路

我制定了"从宏观到微观"的学习路线：先从工程的文件结构和组织架构入手，理清项目骨架；再从 `main.cpp` 入口开始，沿调用链路逐级深入，依次探究各功能模块的逻辑实现。对于过程中遇到的陌生技术栈（如 Qt 信号槽机制、串口通信协议、UDP 网络编程等），同步进行补充学习。

### AI 辅助与知识库建设

学习过程中，我使用 Claude Code 接入 DeepSeek 模型辅助代码分析，所有学习成果沉淀为 Obsidian 知识库。截至目前，知识库包含 **161 篇 Markdown 文档**，按学习阶段分为五大模块：

| 模块 | 内容 | 文档数 |
|------|------|--------|
| 模块化框架理解 | 工程业务流程、线程时序、异常恢复、通信架构等 | 14 篇 |
| 代码导向深读 | 从 main 函数到 GlobalFlow 的完整启动链路逐行解析 | 16 篇 |
| 编译运行实践 | GCC 编译原理、MSVC/MinGW/Linux 三平台编译问题记录 | 8 篇 |
| 界面导向学习 | 双网卡三通道 UDP 通信、界面控件源码溯源、HTTP/MQTT 协议 | 37 篇 |
| MQTT 攻坚 | MQTT 协议原理、QoS、TLS 加密权鉴、嵌入式交叉编译部署 | 15 篇 |

此外，搭建了 **7 个导航索引（MOC）文件**，形成实习生和新员工可用的系统性学习路径，按推荐顺序可于 1-2 周内建立对工程的全面理解。

在使用 AI 的过程中，我持续复盘沟通效果，调整提问策略，生成了多个协作记忆文档，显著提升了后续与 AI 的协作效率。

### 导师反馈与方向调整

导师指出，前期学习存在"过于深入底层"的问题，建议调整策略：先抓住主要框架理解项目整体，后续在负责的具体模块上进行详细深入，做好主次区分。同时，建议搭建编译环境让代码实际运行起来，从另一种视角理解工程的组织架构，接下来以网络通信相关界面和模块为重点。

### 补充实践与创新

根据导师建议，我尝试在多种环境下编译运行工程：

- **Windows + MinGW 4.4 + Qt 4.8.4**：编译通过，运行时崩溃（C++11 代码在 C++98 编译器上的兼容性问题）
- **Windows + MSVC 2010 + Qt 4.8.4**：400+ 级联语法错误，GCC 编译选项泄露至 MSVC，最终放弃
- **Linux Ubuntu 虚拟机**：成功编译运行，仅需修复两处代码逻辑缺陷

将上述三平台编译过程中的所有问题进行了分类汇总，形成《编译错误速查》系列文档。

编译成功后，转向界面导向学习，重点对远程控制、IPC 信息监控、WiFi 配置、系统状态等界面进行了控件到源码的溯源分析，并对其底层通信机制（三通道 UDP、select 非阻塞、环形缓冲队列、线程间信号槽通信）进行了系统总结。

## 工作案例二：网络通信深入学习与 V7.0.9 版本 MQTT 环境配置

### 任务背景

在对工程有了整体了解后，导师安排我开始接触实际工作。提供了一个包含 V7.0.9 版本工程文件的虚拟机，该版本因新硬件升级，部分软件功能的移植和适配尚未完成。需要我完成 MQTT 模块及其权鉴体系的依赖配置，使 MQTT 通信线程能够正常运行。

### 技术方案

1. **环境诊断**：梳理工程中 MQTT 相关源文件（`mqttsrv.cpp`、`mosquitto.h` 等）的依赖关系，识别缺失的动态库、静态库和头文件路径
2. **依赖配置**：在 `.pro` 工程文件中补充第三方库的 include 路径和 link 配置，解开代码中被屏蔽的 MQTT 线程相关段落
3. **编译验证**：完成编译后保存通过的版本，提交至 Git 进行版本管理
4. **功能测试**：搭建本地虚拟 MQTT Broker，解除 PIN 码和 TLS 证书限制，编写测试脚本验证 MQTT 订阅/发布功能正常
5. **交付物整理**：测试通过后回退到编译通过的干净版本，将依赖库的源码、动态库、静态库按 x86（虚拟机）和 ARM（下位机）两种架构分类打包

### AI 辅助经验

在环境配置过程中，AI 能够快速解析编译错误、定位缺失依赖并生成修复方案。但也积累了以下经验教训：
- Agent 操作源码后，须要求其记录详细的修改位置和原因，便于人工确认和回溯
- 做好文件备份和 Git 版本管理，避免 AI 的自动化修改引入难以排查的问题
- Windows 环境下的 Agent 表现明显优于 Linux 虚拟机环境（虚拟机中更易出现幻觉、指令超前等问题），通过建立跨设备的协同记忆文档，有效改善了这一问题
- AI 在编写测试脚本方面效率突出，显著缩短了功能验证周期

### 补充创新

- 将 MQTT 学习过程中的全部知识文档整理为两个子模块：「MQTT 协议详解」和「MQTT 嵌入式设备部署」，归档至知识库
- 通过 Git 和 GitHub 实现多端设备的记忆同步，确保不同工作环境下的 AI 协作连续性
- 对项目介绍文档和记忆文档进行分类梳理，形成了清晰的知识库目录结构

### 验收成果

- 向导师演示了 MQTT 依赖配置完成后，在本地环境与虚拟 Broker 的成功通信，日志中可确认收到订阅 Topic 的消息
- 将全套依赖库文件（源码、动态库、静态库）按架构分类打包提交
