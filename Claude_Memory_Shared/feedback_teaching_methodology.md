# 三层教学法

## 总体原则

1. 用户基础薄弱（无 Qt 经验、无网络知识、无 Linux 多线程经验），需要从底层细节开始，不能跳过基础函数和参数直接讲架构。
2. 用户后续需要在此工程基础上二次开发，需要**细致的理解**而非只记住结论。
3. 所有代码引用必须标明完整工程目录路径（如 `CameraTest(aiuse)/zksort/bus/myudpthread.cpp:430`）。

## 三层结构

### 第一层：基础零件清单（references/ 参考文件）

进入一个模块前，先把涉及的基础类型、函数、类逐一解释，写成独立参考文件。放在 `mdData/references/` 目录下。至少包含：

- `全局参数.md` — 所有 extern 全局结构体（struGsh、struCnfg、struIpcShare 等），标明定义位置、字段含义、读写线程
- `全局单例.md` — 所有全局单例对象（g_Udp、myFlow、g_MainManager 等），标明创建位置、类型、关键方法
- `Qt核心类型.md` — QByteArray、QMap、QMutexLocker、QString、QThread、QTimer 等常用 Qt 类
- `C标准库函数.md` — memcpy、memset、malloc、free、sizeof、printf、sprintf
- `Socket编程基础.md` — sockaddr_in、htons、inet_addr、bind、select、recvfrom、sendto、setsockopt
- `Qt网络类.md` — QUdpSocket、QHostAddress、QSettings

### 第二层：双文档对照输出

每次讲解一个主题时，同时生成两个 MD 文件：

| 文档 | 命名格式 | 内容 |
|------|---------|------|
| 主文档 | `NN-主题名-主文档.md` | 操作流程、代码逻辑、控件溯源。**默认用户已理解其中的基础函数和参数**，不在此文档中反复解释底层细节。使用 Obsidian 的 `[[references/xxx]]` 链接到参考文件 |
| 基础文档 | `NN-主题名-基础知识.md` | 主文档中出现的所有基础概念、函数、类型、专有名词，**按在主文档中首次出现的顺序排列**，用户可对照阅读 |

### 第三层：界面控件溯源

回到界面，逐个控件回答：

1. **创建位置**：哪个文件/哪一行创建了这个控件
2. **触发的信号/槽**：操作控件时触发什么函数
3. **函数详情**：入参是什么、出参是什么、做了什么
4. **背后线程**：涉及哪些线程、线程间如何通信、数据如何从子线程回到主线程

## 文档命名规范

- 项目文档（学习计划、设计文档等）的文件名统一使用**中文命名**
- 所有 MD 文档存放在 `Desktop/mddata/` 下
- 参考文件存放在 `mddata/references/` 子目录下
- 使用 Obsidian 的 `[[文件名]]` 语法做交叉链接

## 框架优先策略（针对完全陌生领域）

当用户对某个领域完全零基础（如加密、网络协议栈）时，**先给概念框架地图，再按需深入**。不要上来就给完整的主文档+基础知识双文档。

步骤：
1. 先给 5-7 个核心概念和它们之间的关系图（在对话中，不写文件）
2. 提供学习路线建议（按依赖关系排序）
3. 用户确认方向后，再展开具体内容
4. 展开完成后统一导出 MD

**Why:** 本阶段加密学习验证——用户说"加密我一窍不通，甚至不知道怎么提问"。先给框架地图后，用户能主动导航学习方向。全陌生领域跳过框架直接给文档会导致信息过载。

**How to apply:** 当用户说"XX领域我完全零基础"时，使用此策略。

## 文档脉络与章节承接

撰写学习类文档时，必须有**一条清晰的主线和框架**，做好每一章之间的承接描述。文档不是一个知识点列表——它是带逻辑递进关系的教程。

核心要求：

1. **明确主线**：文档开头说明"这条主线是什么，从哪开始，到哪结束"
2. **章节间承接**：每一章结尾交代"这一章和下一章的关系是什么，为什么学了这章之后要学下一章"
3. **依赖说明**：如果章节 B 依赖章节 A 的概念，在章节 A 结尾或章节 B 开头明确说出这种依赖关系

承接示例（差 vs 好）：

```
差："第一章 MQTT 报文格式。第二章 TLS 加密。第三章 本工程代码。"
    ↑ 章节之间没有连接，读者不知道为什么这个顺序，不知道为什么加密突然出现

好："上一章我们搞清楚了 MQTT 报文在 TCP 字节流中怎么划分边界。
     但所有这些报文都是明文传输的——任何人抓包都能看到 JSON 内容。
     所以这一章我们来看 TLS 是怎么在 TCP 和 MQTT 之间插入一层加密的。"
    ↑ 解释了为什么从报文跳到加密，两者的关系是什么
```

**Why:** 用户基础薄弱，面对切割感强的文档时，需要自行脑补章节之间的逻辑关系——这件事对新手来说非常困难。章节承接就是替用户完成"为什么下一章是这个"的逻辑推导。最近的 MQTT 文档在这方面做得很差——MQTT 协议 → MQTT 加密 → 本工程代码之间缺乏连贯性，导致阅读痛苦。

**How to apply:** 撰写任何多章节学习文档时，每写完一章检查：
- 下一章和这一章的关系是否明确说明了？
- 读者是否知道"为什么这个顺序"而非别的顺序？
- 两章之间是否有"过渡句"？

## 深度广度学习模式

当用户说"使用深度广度学习模式"或"深度广度模式"时，不直接开始讲解。先完成两个确认步骤：

### 第一步：广度罗列

把要讲解的主题拆解为树状的知识点列表，展示给用户。格式为：

```
- 主题A
  - 子知识点A1
  - 子知识点A2
- 主题B
  - 子知识点B1
  ...
```

**主动补充**：如果我认为某些用户没提到的知识点和当前主题有逻辑关联（前置依赖、自然延伸、对等对比），主动指出："你提到的XXX，和YYY有[关系类型]，是否需要纳入广度范围？"

### 第二步：深度确认

描述我理解的要讲到的深度层次，例如：

"深度：从[起点概念]开始，讲到[终点]，核心函数/机制拆解到[具体层次，如系统调用/内核数据结构/Qt源码级别]"

用户确认广度和深度后，才开始生成内容。

**Why:** 用户的学习模式是"广度遍历→找关系→梳理逻辑→按框架深挖"。这个模式把前两步（广度遍历、找关系）前置到讲解开始之前，避免讲完之后用户才发现"这没覆盖"或"这太浅/太深"。加密学习阶段"不知道怎么提问"的问题本质上是广度没展开——用户不知道这个领域有哪些概念可以问。

**How to apply:** 用户明确说"深度广度模式"时触发此流程。日常简单问答不需要。

**重要**：深度广度模式的最终产出必须导出 MD 文档，用户确认广度和深度后，在生成内容的同时写入 md 文件。

**禁止打比方、比喻、类比。** 不使用"就像快递单一样""好比xx""相当于xx"。用数据结构、代码逻辑、调用关系直接解释，从底层数据结构角度说明。

**Why:** 用户要求深入底层的技术学习，比喻和类比增加了理解负担而非降低难度。用户需要的是精确的技术术语和直接的数据结构描述。

**How to apply:** 解释任何概念时直接描述其数据结构（成员变量、方法、调用链），不使用生活类比。如果需要对照，只使用用户已知的技术概念（STM32、C++、Qt 已学过部分）。

- 用户说"导出md"时才写文件，日常解释在对话中完成
- 每个主题结束后主动询问用户是否继续下一步
- 用户提出错误时，立即承认并修正（如之前的远程控制页面搞混）

**Why:** 用户之前的学习方式（架构先行、跳过底层细节）导致只能一知半解记住结论。调整为底层→上层、零件→整体的教学顺序后效果显著提升。

**How to apply:** 当用户说"按照记忆中的教学方式来讲解 XX"时，严格按照本文件的三层结构执行。先确保 references/ 参考文件齐全，然后生成主文档+基础知识文档，最后溯源界面控件。
