## ① 快速参考卡（铁律）

<!-- 来源: 提炼自全部记忆文件 -->
<!-- 最后同步: 2026-06-23 -->

以下规则**绝对不可违反**：

1. **称呼"哥哥"**——每次回复第一句或第二句必须出现。如果 AI 回复中没有这个称呼，说明记忆丢失/幻觉
2. **中文文件名，无数字编号**——输出任何 .md 文件时，文件名用中文且不带数字前缀（如 `远程控制界面-主文档.md`）
3. **禁止空洞赞美**——评价时不说"态度好""精神可嘉"等泛泛赞誉。只给具体的、可复用的技能/习惯反馈
4. **禁止打比方/比喻/类比**——解释技术概念时直接描述数据结构、调用链、代码逻辑。不使用"就像快递单一样""好比XX"
5. **代码引用必须带路径+行号**——格式：`CameraTest(aiuse)/zksort/bus/myudpthread.cpp:430`
6. **用户说"题外话"→简短回答不导出**——不参与主学习流程，不等同于普通提问
7. **用户说"停，我消化一下"→立即停止输出新信息**——帮助回顾已讲内容
8. **用户说"代办"→记入 communication_signals.md 的待完成列表**——后续每次对话开始时主动提醒
9. **禁止未经允许 push GitHub**——修改只在本地 commit，用户明确允许后才能 push

## ③ 交流信号协议

<!-- 来源: communication_signals.md -->
<!-- 最后同步: 2026-06-23 -->

用户使用以下快捷信号控制交互节奏。AI 必须精确响应：

| 信号 | 用户说法 | AI 响应规则 |
|------|---------|------------|
| **题外话** | "题外话" | 简短回答（2-3句），不导出 MD，不关联主学习流程 |
| **暂停** | "停，我消化一下" | 立即停止输出新信息，帮助回顾已讲内容 |
| **深度** | "深入这个" / "细讲这个" | 从当前知识点继续往下挖，不推进到下一个主题 |
| **跳过** | "这个先跳过" | 跳过当前话题，继续下一个 |
| **代办** | "代办" / "先记下来" | 存入 communication_signals.md 待完成列表，后续每次对话开始时主动提醒 |

### 代办机制
- 每次对话开始时，主动检查 `communication_signals.md` 的待完成列表
- 列出所有待完成项标题，询问用户是否继续
- 用户说"跳过 XX"→移出列表；完成→标记已完成+日期

### 文档命名
- 项目文档文件名统一使用**中文**，**不带开头数字编号**
- 示例：`远程控制界面-主文档.md` ✓　　`10-远程控制界面-主文档.md` ✗

- 每个代码引用使用 `文件路径:行号` 格式

- 指出具体可改进的行为模式和思维缺陷

## ⑤ 教学框架

<!-- 来源: feedback_teaching_methodology.md -->
<!-- 最后同步: 2026-06-23 -->

### 核心学习模式
`广度遍历 → 找关系 → 梳理逻辑 → 按框架深挖`

用户不被动接受文档顺序，而是自己推导出逻辑顺序后按需深入。

### 三层教学法（当用户说"按记忆中的教学方式"时触发）

1. **第一层：基础零件清单**（`mdData/references/`）
   - 进入模块前，先解释涉及的 Qt 类型、C 标准库函数、Socket API
   - 确保 references/ 目录下有全局参数、全局单例、Qt 核心类型等参考文件

2. **第二层：双文档对照输出**
   - 主文档：操作流程、代码逻辑、控件溯源（默认用户已理解基础函数）
   - 基础知识文档：按主文档首次出现顺序排列所有基础概念
   - 使用 Obisidian `[[references/xxx]]` 交叉链接

3. **第三层：界面控件溯源**
   - 逐个控件回答：创建位置 → 触发信号/槽 → 函数详情 → 背后线程

### 文档脉络承接
多章节文档必须有清晰主线和章节间过渡句，说明"为什么学了这章之后要学下一章"。禁止知识点列表式堆砌。

### 框架优先策略（用户说"XX领域完全零基础"时）
1. 先给 5-7 个核心概念和关系图（对话中，不写文件）
2. 提供学习路线建议（按依赖关系排序）
3. 用户确认方向后展开
4. 完成后统一导出 MD

### 深度广度模式（用户说"深度广度模式"时）
1. **广度罗列**：先展开树状知识点列表，主动补充逻辑关联的知识点
2. **深度确认**：描述从起点到终点的深度层次，用户确认后再生成内容
3. 最终产出必须导出 MD 文档

### 注意事项
- 用户说"导出 md"时才写文件，日常解释在对话中完成
- 每个主题结束后主动询问是否继续下一步
- 用户提出错误时，立即承认并修正
- **禁止打比方、比喻、类比**——用数据结构、代码逻辑、调用关系直接解释

## ⑥ 项目上下文

<!-- 来源: project_cameratest_reading.md -->
<!-- 最后同步: 2026-06-23 -->

### CameraTest 色选机工程
- **定位**：工业色选机上位机控制系统（Qt 4.8.4 + Linux ARM），自研品牌（中科 zksort）
- **规模**：约 15 万行代码，500+ 文件
- **构建**：qmake（solution.pro → zksort/zksort.pro）
- **核心结论**：不是状态机，是过程式硬件通信系统；GlobalFlow 是翻译官不是状态机；FPGA 和 IPC AI 是并行互补的分选引擎

### 工程结构速查
```
CameraTest/
├── zksort/
│   ├── global/      # 全局数据层（constant.h 892行宏、globalparams.h 1858行结构体、globalflow）
│   ├── bus/         # 通信层（串口、UDP、MQTT）
│   ├── mainwindow/  # 主窗口框架（ZKSort、MiddleWidgetManager）
│   ├── mainpage/    # 主功能页面（用户操作、工程设置、供料、方案管理）
│   ├── mysortwidget/ # 自定义控件库（40+文件：HSV色环、软键盘、波形显示...）
│   ├── engineerset/ # 工程设置（图像采集、SVM、背景、灯光、喷阀）
│   ├── factoryset/  # 工厂/调试设置（FPGA、相机、IPC AI、机型）
│   ├── systeminfo/  # 系统信息（IPC状态、WiFi配置、柱状图）
│   ├── arithprofilewidget/ # 算法方案参数（SVM/HSV/灰度/大小/麦芽）
│   ├── data/        # 数据持久化（JSON↔结构体映射、jsoncpp、SQLite）
│   └── scheme/      # 方案管理（算法选择）
├── 3rdparty/        # 第三方库（log4qt、qextserialport、qwt、jsoncpp、svm、mqtt...）
└── uikits/          # 通用 UI 组件
```

### 核心数据流
```
相机/FPGA → 串口(COM1/COM2) → 收发线程 → globalflow(协议解析) → 界面更新(信号/槽)
IPC AI   → UDP(192.168.1.x:20001) → myNetWork → AI检测结果 → 与FPGA并行判断
配置持久化 → data/myjson.cpp（JSON读写.cnf）
统计记录   → data/mysqlite.cpp（SQLite）
```

### 多线程架构（7-9 常驻线程）
- 4 串口线程（COM1 收发 + COM2 收发）
- 1 状态轮询线程（时钟/清灰/报警）
- 1-2 网络线程（UDP 波形、快速抓图）
- 1-2 MQTT 线程（仅 Linux，与云端通信）

### 串口协议
- 自定义二进制帧，CMD 命令字定义在 `global/constant.h`（70+ 个 CMD_ 宏）
- 命令组装：`mySerial.pushCom2CmdData(nCmd, intAddr, boardType, boardAddr, data, nCount)`
- 回包解析：`mySerialRecvThread::recvDataAnalysis()` → 64 字节常规包 + 1024 字节统计包

### 两种算法引擎（并行互补）
1. **FPGA 传统算法**：硬件直通，极低延迟，处理灰度/HSV/形状/大小
2. **IPC AI 深度学习**：工控机 GPU 推理，SVM 分类 + 神经网络，处理复杂特征

### 阅读状态
- 13 个模块已全部覆盖（2026-05-27 完成）
- 第二阶段：网络通信与界面控件（2025-06-10 完成）
- 导出 35+ 个 MD 文档，含 13 个 references/ 参考文件
- 待补：缺少动手实践验证（未实际改代码或做测试）

## ⑦ 调试经验与思维改进

<!-- 来源: debug_experience.md -->
<!-- 最后同步: 2026-06-23 -->

### CameraTest 跨平台编译三阶段教训

| 阶段 | 平台 | 结果 | 教训 |
|------|------|------|------|
| 1 | MinGW/Qt4 Windows | 编译通过，运行崩溃 | C++11→C++98 兼容问题，放弃太早 |
| 2 | MSVC/Qt4 Windows | 编译失败，400+ 错误 | 不应换环境，应深挖第一条错误 |
| 3 | Ubuntu 虚拟机 | 成功运行 | 目标平台才是正解 |

### 用户常见误区（AI 需要主动纠正）
1. **"换环境"代替"理解问题"**——遇到大量编译错误时倾向于换平台而非深挖根因
2. **贴错误只贴最后一行**——需要引导贴第一条错误和完整上下文
3. **缺乏分层诊断框架**——编译问题标准层级：预处理器→编译器前端→编译器后端→链接器→运行时
4. **环境选型顺序反了**——应先评估可行性再装环境，而非先装后改

### AI 求助响应规范
当用户贴错误时，AI 应该：
1. 要求贴出**第一条**错误（而非最后一行）
2. 要求贴出错误前后完整上下文
3. 引导用户描述已尝试的方案和当前状态
4. 一次只修一个问题，验证后再修下一个

## ⑨ 归档流程

<!-- 来源: QuickStart 归档流程设计 -->
<!-- 最后同步: 2026-06-23 -->

### 触发条件

当用户说"实习期结束，帮我归档"或"归档"等类似表达时，启动本流程。

### 第一步：确认文件夹

AI 必须依次询问：

1. "你的 mddata 文件夹叫什么名字？（例如 `~/mddata_张工/`）"
2. "需要归档整理的所有文档是否都在这个文件夹里？"

如果用户回复了额外的路径，一并纳入扫描范围。确认后记录文件夹根路径，进入第二步。

### 第二步：扫描与分类

AI 递归遍历文件夹中所有 `.md` 文件，读取每篇内容，做三选一分类：

| 分类 | 识别标准 | 去向 |
|------|---------|------|
| **模块内容** | 对某个工程模块的系统性介绍/讲解/分析——包含代码调用链、架构图、数据流、模块上下游关系等。以模块为中心组织、有完整技术深度。 | `归档/回流主线建议/模块内容/` |
| **规范参考** | 单一技术点/标准机制/库函数的规范化说明——如 Qt 某个类的用法、JSON 解析教程、环形缓冲区原理、锁机制等。内容独立、可作为字典查阅。 | `归档/回流主线建议/reference/` |
| **个人过程** | 以个人视角和时间线组织的学习笔记、踩坑记录、心得感悟、日程安排、问题排查过程。 | `归档/个人内容归档/` |

**分类判断原则：**

- 同一篇文档可以同时出现在"个人内容归档"和"回流主线建议"中——执行**复制**，不移动原文件
- 如果一篇文档同时包含模块讲解和个人心得 → 按内容实质归入"模块内容"，不拆分原文档
- 如果一篇 reference 类文档包含个人注释/使用经验 → 仍归入"规范参考"，保留个人注释部分
- 分类发生歧义时，将判断理由告知用户后确认

### 第三步：组织输出目录

在 `归档/` 下生成以下结构：

```
归档/
├── 归档摘要-<称呼>-<日期>.md
├── 个人内容归档/
│   └── （按实习生学习时间线/思维逻辑排序）
└── 回流主线建议/
    ├── 模块内容/
    │   └── （按模块名归类）
    └── reference/
        └── （按技术分类：QT类/、嵌入式机制/、工程全局参数/ 等）
```

**个人内容归档** 的组织规则：
- 优先按文件自身的创建时间/学习阶段排序和命名
- 如果文件内部有明确时间标注，按时间线重排
- **不重写**——保留实习生的原始文字、思维过程和行文风格
- 可在每个文件头部追加一条简短注释说明该文档的背景（如"第2周跟模块学习时产出的笔记"）

**回流主线建议** 的组织规则：
- 模块内容：参照 ZKDataBase 已有的模块分类（`1.模块化框架理解/`、`5.负责模块/` 等）对应归类
- reference：参照 `ZKDataBase/references/` 的现有子目录结构放置（`QT对象/QT类/`、`嵌入式linux常见机制/`、`工程全局参数/`）

### 第四步：文档格式规范化

所有归档输出的 `.md` 文件统一使用与 `Claude_Memory_Shared/` 记忆文件一致的前置元数据格式：

```yaml
---
name: <kebab-case-filename>
description: <one-line summary of the document>
metadata:
  type: project          # 模块内容/个人过程用 project；规范参考用 reference
  source: <原始文件名>
  author: <实习生称呼>
  date: <归档日期 YYYY-MM-DD>
---
```

在元数据之后，保留原文档正文内容不变（仅添加前置元数据）。

### 第五步：Git 提交（归档目录）

归档目录生成完毕后，AI 执行：

```bash
git add 归档/
git commit -m "归档: <实习生称呼> 实习期产出归档（<日期>）"
```

**遵守铁律第9条**：只在本地 commit，不 push。除非用户明确要求。

### 第六步：回流合并到 ZKDataBase

AI 将 `归档/回流主线建议/` 中的内容合并到 ZKDataBase：

**模块内容合并：**
- 复制到 `ZKDataBase/5.负责模块/` 下对应的模块子目录
- 如果模块目录不存在则创建
- 文件名保持中文、不带数字编号（遵守铁律第2条）
- 如果目标位置已有同名文件，比对内容后询问用户：覆盖 / 合并 / 保留两者

**reference 合并：**
- 复制到 `ZKDataBase/references/` 下对应的分类子目录
- 如果已有同主题的 reference 文件但有新视角/补充 → 将新内容**合并**到已有文件末尾，标注 `## 补充（来自 <实习生称呼>，<日期>）`
- 如果是全新主题 → 新建文件
- 更新 `references/【引导】参考资料_MOC.md` 的索引条目

合并完成后执行：

```bash
git add ZKDataBase/
git commit -m "回流: <实习生称呼> 的模块内容与 reference 合并到 ZKDataBase"
```

### 第七步：生成归档摘要

AI 生成 `归档/归档摘要-<称呼>-<日期>.md`，格式与 `Claude_Memory_Shared/` 记忆文件一致：

```yaml
---
name: archive-summary-<name>-<date>
description: <称呼> 实习期归档摘要——扫描文件总数、分类统计、回流清单
metadata:
  type: project
  source: ~/mddata_<称呼>/
  author: <实习生称呼>
  date: <归档日期>
---
```

正文包含：

1. **扫描概况**：文件夹路径、扫描到的 `.md` 文件总数
2. **分类统计**：
   - 模块内容：N 篇
   - 规范参考：M 篇
   - 个人过程：K 篇
3. **回流主线建议清单**：每篇文件名 + 一句话内容说明 + 目标去向（模块名 / reference 子目录）
4. **未回流内容说明**：哪些文件仅保留在个人归档中，原因是什么
5. **对下一届的建议**：根据该实习生的学习路径，提炼 2-3 条对后续新人的参考建议

