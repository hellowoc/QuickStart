# AI 反馈偏好

## 一、文档命名规范

项目文档（学习计划、设计文档等）的文件名统一使用中文命名，且**不带开头数字编号**。如 `远程控制界面-主文档.md` 而非 `10-远程控制界面-主文档.md`。

**Why:** 中文文件名更直观易读。数字编号在 Obsidian 中无意义，且后续增删文档时编号会乱。

**How to apply:** 在 memory_claude 项目中创建任何文档类 .md 文件时，文件名使用中文且不带数字前缀。导出到 mdData/ 的文档同样遵守。

## 二、代码讲解详细度

讲解代码时须标明所在文件夹和文件路径，描述 UI 控件时须说明：控件在哪个文件中创建、相关函数在哪、完成什么功能、背后有什么进程/线程。

**格式要求：**
- 每个代码引用使用 `文件路径:行号` 格式
- 每个控件描述格式：创建位置 → 关联函数 → 功能 → 背后线程/进程

**Why:** 用户没有网络基础和 Qt 经验，需要精确的代码定位来帮助建立代码-界面-行为的映射关系。

## 三、评价风格

评价时不要出现为赞美而赞美的表述（如赞扬精神、意志力、态度等意识形态层面）。只给出绝对客观、对个人能力提升有实际作用的评价，指出具体可改进的行为模式和思维缺陷，避免任何主观过誉。

**How to apply:** 在任何需要评价用户行为、思维模式、工作方法的场景中，跳过"态度很好""精神可嘉"等表述，直接进入具体问题分析和改进建议。优点也应限于可复用的、具体的技能或习惯，而非泛泛的品质描述。
