# ZKDataBase 三区知识库结构设计

## 背景

ZKDataBase 需要支撑多人协同积累知识：
- 一个权威主线保证方向正确
- 个人自由空间保留学习特色
- 好的个人产出自然升级为公共知识

## 三区架构

```
mdData/
├── _主线_/          ← L1 权威参考区
├── _模块_/          ← L2 协同贡献区
├── _个人_/          ← L3 自由空间区
├── _模板_/          ← 通用模板
└── README.md        ← 新人入口
```

## 各区规则

| 区域 | 谁写 | 审核 | 性质 | 命名要求 |
|------|------|------|------|---------|
| `_主线_/` | 维护者 | 自己把关 | 权威参考，新人首先阅读 | 无前缀，如 `工程概述.md` |
| `_模块_/` | 任何人 | 维护者简单过目 | 模块深入解读 | `_权威-`(维护者) 或 `署名-` |
| `_个人_/` | 自己 | 无 | 自由空间 | 无要求 |
| `_模板_/` | 维护者 | 自己 | 参考模板 | `模板-` 前缀 |

## 知识流转规则

```
_个人_/张三/学习笔记.md
    │
    │ 张三觉得有价值的理解
    ▼
_模块_/MQTT通信/张三-心跳超时分析.md   ← 署名贡献
    │
    │ 维护者审核，认为够权威
    ▼
_主线_/通信层-MQTT全链路.md            ← 融合进主线
```

## 权威前缀说明

- `_权威-xxx.md`：维护者写的模块延伸文章，与主线等权威
- `署名-xxx.md`：个人贡献，注明作者，读者自行判断可信度
- 同一主题允许多个署名文章并存，不同角度互补

## 对比原有结构

| 原有结构 | 问题 | 三区方案 |
|---------|------|---------|
| `1.模块化框架理解/` | 数字编号，谁定的顺序？ | 归档到 `_主线_/`，文件名不编号 |
| `2.代码导向的工程学习/` | 单人视角 | 多人可往 `_模块_/` 补充 |
| `5.负责模块/` | 和通用知识混在一起 | 拆入 `_个人_/张三/负责模块/` |
| `references/` | 基础零件散落 | 归入 `_主线_/references/` |
| `私有库构建/` | 位置不明确 | 归入 `_个人_/` |
